Tuple

タプルは

  • イミュータブル
  • 順序付き
  • 作成時にサイズ固定
  • 異種混合（異なる型の値を含めることができます）
let ageAndName = (24, "Lil' Reason");
let my3dCoordinates = (20.0, 30.5, 100.0);

タプルの型は型注釈でも使用できます。タプル型は視覚的にタプルの型に似ています。

let ageAndName: (int, string) = (24, "Lil' Reason");
/* タプル型エイリアス */
type coord3d = (float, float, float);
let my3dCoordinates: coord3d = (20.0, 30.5, 100.0);

備考: サイズが1のタプルはありません。単に値自身を使います。

使い方

標準ライブラリは fst と snd （ここの、 "Pair operations" の下）を提供し、2組のタプルの第1要素と第2要素にアクセスするための便利な関数です。 一般的に、デストラクチャリングを通してn組のタプルのメンバにアクセスします（サイドバーの後述のように）:

let (_, y, _) = my3dCoordinates; /* y がバインドされました */

_ は、対象のメンバを無視していることを意味します。

タプルは突然更新されることはありません。古いものを破棄することで、新しいものを作ります。

ヒントとコツ

形式張らずに複数の値を渡すことが役に立つ状況でタプルを使います。例えば、多くの値を返す時:

let getCenterCoordinates = () => {
  let x = doSomeOperationsHere();
  let y = doSomeMoreOperationsHere();
  (x, y)
};

あるいは、可能性の組み合わせの「パターンマッチ」（後述）:

switch (isWindowOpen, isDoorOpen) { /* これは2組のタプルです */
| (true, true) => ...
| (true, false) => ...
| (false, true) => ...
| (false, false) => ...
}

タプルの使用は局所的に保つようにしてください。長い間頻繁に使いまわすデータ構造なら名前付きフィールドを持つ、レコードが適します。

タプル型は「積型」とも呼ばれ、(string, int) は、string * int で書かれています。 タプルは実際には「デカルト積」であるという考えがあります。x軸にstring、y軸にintを持つ2Dグリッドを想像してみてください！

タプル + switch の組み合わせは非常に強力かつ簡潔で、バグのカテゴリ全体を一掃します。 それと共に、それらはきれいにすべての可能な値の組み合わせを列挙します。 (bool, bool) 型のタプルは確かに 2 * 2 = 4の可能性を持ち、型システムはあなたにすべての 4 をカバーすることを要求します。 これはコードのリファクタリングに役立ちます。任意の値のあちこちにいくつかのif-elseを非形式的に追加する代わりに、あなたが変更を好みます必要とする正確なswitch 分岐を直接特定することに他なりません。

設計方針

型のない言語から来た人にとってタプルの存在は奇妙に見えるかもしれません。「なぜリスト/配列を使用しないのですか？」

型システムはすべてにおいて強力なわけではなく、また、そうでなくてはなりません。言語をシンプルかつ、パフォーマンス（コンパイルと実行速度の両方）と理解しやすさを保つために、いくつかのトレードオフが適用される必要があります。 例えば、Reason のリストは、サイズがより柔軟で、連結、追加、スライスなどが可能です。 その代わり、一様（リストごとに1種の型の値しか含めることができない）であることが求められ、ランダムインデックスアクセスは常に有効であるとは限りません。 一方、タプルは、サイズに対する制約により、より速く、型システムにすべての要素の型を徹底的に追跡し、安全なアクセスを保証する余裕を与えます 一般的に、型システムの幾つかの主要な、味わい深い、トレードオフに気づくでしょう: レコードのフィールドは固定される代わりに、多様（複数の型を許容する）で、マップのフィールドは柔軟な代わりに、一様（複数の型を許容しない）でなければならない、など。

Reason の型は「構造的に」型付けされます。 これは、たとえあなたがデータに明示的な型注釈をしなくても、コンパイラが内容や使われ方などから判断して、推論することも可能であることを意味します。 宣言と用途の推定された形が一致する限り、良いことしかありません！

Reason の型システムは多様な動的サイズのリストを許可できませんが、実際には可能（ヒント: GADT）だったりします！ しかし、そのような機能をデフォルトにすることは、初見の学習オーバヘッドとコードの理解しやすさを増加させます。 型がそれを達成できるというだけで、コードの一部を無限に複雑にすることは、常に良い考えであるとは限りません！
