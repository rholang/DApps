Pattern Matching!

Make sure you've read up on Variant first.

とうとうやってきました！パターンマッチングはこの言語のザ・ベスト機能のひとつです。これは構造化代入に似ていますが、型システムの恩恵をより受けられます。

使い方

ヴァリアントを考えてみましょう。

type payload =
  | BadResult(int)
  | GoodResult(string)
  | NoResult;

switch 式の中で、それを「構造分解」することができます。

let data = GoodResult("Product shipped!");

let message =
  switch (data) {
  | GoodResult(theMessage) => "Success! " ++ theMessage
  | BadResult(errorCode) => "Something's wrong. The error code is: " ++ string_of_int(errorCode)
  };

それぞれの異なるケースを処理している間、data をどのように構造分解したか注目してください。 上記の switch はコンパイラの警告を出力するでしょう。

Warning 8: this pattern-matching is not exhaustive.
Here is an example of a value that is not matched:
NoResult

すごくないですか？ 型システムはあなたのデータの形をマッチングしつつ、未処理のケースについて警告しました。 この条件的側面は、単純な構造化代入ではなく、パターンマッチングを行うものです。 「これならこれ」といった体のほとんどのデータ構造で動きます。

switch (myList) {
| [] => print_endline("Empty list")
| [a, ...theRest] => print_endline("list with the head value " ++ a)
};

switch (myArray) {
| [|1, 2|] => print_endline("This is an array with item 1 and 2")
| [||] => print_endline("This array has no element")
| _ => print_endline("This is an array")
};

最後の _ のケースは、マッチしないすべて条件をその分岐に行くようにするための特別なフォールスルーケースです。

string、int、その他も switch できます。同じ結果に行きつく多くのパターンを持つこともできます！

let reply =
  switch (message) {
  | "Reason's pretty cool" => "Yep"
  | "good night" => "See ya!"
  | "hello" | "hi" | "heya" | "hey" => "hello to you too!"
  | _ => "Nice to meet you!"
  };

他のデータ構造と組み合わせることで、パターンマッチングは非常に簡潔で、コンパイラによって検証された、パフォーマンスの高いコードを作り出すことができます。

let message =
  switch (data) {
  | GoodResult(theMessage) => "Success! " ++ theMessage
  | BadResult(0 | 1 | 5) => "Something's wrong. It's a server side problem."
  | BadResult(errorCode) => "Unknown error occurred. Code: " ++ string_of_int(errorCode)
  | NoResult => "Things look fine"
  };

注: パターンとしてリテラル（つまり具体的な値）のみ渡すことができ、名前や他のものを let 束縛することはできません。下記は期待通りに動作しません。

let myMessage = "Hello";
switch (greeting) {
| myMessage => print_endline("Hi to you")
};

代わりに、上記は期待する結果にはならず、switch のケースの中で、任意の文字列にマッチングして、それに myMessage という名前を束縛することになるでしょう。

When 節

完全なパターンマッチで任意のロジックを本当に使いたいときは、when 節に流すことができます。これは基本的には if のシュガーです。

let message =
  switch (data) {
  | GoodResult(theMessage) => ...
  | BadResult(errorCode) when isServerError(errorCode) => ...
  | BadResult(errorCode) => ... /* otherwise */
  | NoResult => ...
  };

例外にマッチする

もし関数が例外（後で説明します）をスローするなら、関数は通常に返される値に加えて、それ（例外）にマッチすることもできます。

switch (List.find((i) => i === theItem, myItems)) {
| item => print_endline(item)
| exception Not_found => print_endline("No such item found!")
};

入れ子パターン

入れ子になった | は意図どおり動作します。

switch (student) {
| {name: "Jane" | "Joe"} => ...
| {name: "Bob", Job: Programmer({fullTime: Yes | Maybe})} => ...
};

パターンはどこにでも

パターンは通常の「変数宣言」を書くであろう箇所のどこにでも書けます。

type leftOrRight =
  | Left(int)
  | Right(int);

let i = Left(1);

/* magic! */
let Left(v) | Right(v) = i;

ヒントとコツ

可能な限りパターンマッチをフラット化してください。これは実際にバグの除去装置です。以下で例示します。

フォールスルーの _ ケースを乱用しないでください。 これはコンパイラがあるケースをカバーすること（網羅性チェック）し忘れていると助言してくれることを妨げてしまいます。それはあなたがヴァリアントに新しいケースを追加するリファクタリングの後、特に有用でしょう。 無限の可能性があるパターン（例: string, int など）に対してのみ _ を試してください。

最悪から最高までの一連の例がここにあります。

let optionBoolToBool = opt =>
  if (opt == None) {
    false;
  } else if (opt == Some(true)) {
    true;
  } else {
    false;
  };

今が最悪です =)。パターンマッチングに切り替えてみましょう。

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(a) => a ? true : false
  };

少しマシですが、まだ入れ子です。パターンマッチングはこうすることを許容します。

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(true) => true
  | Some(false) => false
  };

かなり線形な見た目です！今、あなたはこうしたいと駆られているかもしれません。

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(true) => true
  | _ => false
  };

Which is much more concise, but kills the exhaustiveness check mentioned above; refrain from using that. This is the best:

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(trueOrFalse) => trueOrFalse
  | None => false
  };

この時点でこのコードで間違いを犯すのはかなり難しいです！ 多くの分岐で if-else を使用したい時はいつでも、代わりにパターンマッチングを選びます。 それはより簡潔で高性能でもあります。

別の例を参照してください、switch + タプルの例はここです。

設計方針

悪名高き fizzbuzz プログラムは、いくつかの人々を妙に躓かせます。どこにもないエレガントな探索でいくつかの条件分岐を簡潔化/統一化すること望むプログラマーを麻痺させる性質があるためです。 うまくいけば通常、パターンマッチングの視覚的簡潔性は、総当りの if-else の利点のすべてを享受しつつ、決定の麻痺を克服させてくれます。 実際にすべての可能性を明示的に一覧することは間違いはありません。パターンマッチングは妥当な問題解決テクニックとして非常に便利であることが知られている数学の場合分けに対応します。

初めて Reason の switchを使うと、何年も過ちを犯していたのではないかと感じるかもしれません。あなたにとって他の言語が台無しになってしまうかもしれないことに注意してください =)。

もし大きなリファクタリングをしようとした場合、if-else のロジックが入れ子になっていたら、そのロジックの正しさを維持することは難しいと気がついたかもしれません。 その一方で、パターンマッチング + タプルは2次元表にマッピングされ、各マスは独立して埋められます。 これにより switch にケースを追加する必要がある時はいつでも、他のセルを乱すことなく、そのテーブルセルのみを対象にできます。

type animal = Dog | Cat | Bird;
let result = switch (isBig, myAnimal) {
| (true, Dog) => 1
| (true, Cat) => 2
| (true, Bird) => 3
| (false, Dog | Cat) => 4
| (false, Bird) => 5
};
