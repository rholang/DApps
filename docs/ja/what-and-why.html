What & Why

Reason とは？

Reason は新しい言語ではありません。百戦錬磨の言語、OCaml で動く新しい構文とツールチェインです。 Reason は OCaml に JavaScript プログラマーが慣れ親しんだ構文を与え、既存の NPM/Yarn のワークフローを使っているプログラマーに応えます。

つまりは、Reason は、強く静的に型付けされた、より速く、よりシンプルな JavaScript の親戚で、今日の ES から負の遺産を引いて、ES2030 の機能を足したもので、JS と OCaml の両エコシステムからアクセスできるツールとみなせます！

Reason は OCaml/Reason を相互運用し易いリーダブルな JavaScript にコンパイルする姉妹プロジェクトの BuckleScript の力を借りて JavaScript にコンパイルされます。 Reason は OCaml 自体のおかけでネイティブコードにもコンパイルされます。

なぜ Reason なのか？

「どうして新しい言語それ自体を学ばなければならないのか？」

Reason に関して言えばそれは違います。

実際のところ、すでに OCaml のセマンティクス（実行方法）の 80% は、モダンな JavaScript に直にマッピングできます。逆もまた然り。贅沢にも JavaScirpt の微妙な機能をいくつか外して、ナイスな機能をいくつか追加したとしても、実際にはリーダブルな JS にコンパイルされ、それらのエコシステムとツールの 80％ を直接使用することができると言うことです。 しかも、ネイティブコード、iOS、Android、そして マイクロコントローラ向けにさえもコンパイルできる！

しかし、高速なセマンティクスと完全な型付けをもたらすために、JS のどの機能を再構成するかははっきりしていません。 ただ、私たちは、すでにパフォーマンスと簡潔さを兼ね備えている健在な言語から、見た目や動作をいくらか微調整して、使い慣れた Web 言語のより良い選択肢となるように逆算しながら作業することができます。

これらの決定は Reason の学習曲線を JS + 漸進的型システムを学ぶよりも緩やかにすることをもたらしました。その見返りとしてあなたが得られるのは:

  • 厳格で強い型システム。 Ocaml の型は 100% のカバレッジ（コード全行）、型推論（型は推論され手動で記述する必要はありません）、安全性（一度コンパイルされると、型が正しいことが保証されます）を持ちます。
  • 簡潔な見た目 & 実用主義。 言語の残りの部分を純粋でイミュータブルかつファンクショナルに保ちつつ、副作用、ミューテーション、親しんだオブジェクトとの相互運用性を許容します。
  • パフォーマンス & サイズにフォーカス。 Reason のビルドシステムである bsb は、100ms（増分）未満でビルドを完了します。 私たちの作品のアウトプットも小さいです。
  • インクリメンタルな学習 & コードベース変換。 初日から完全に型付けされたファイルの恩恵を受けてください。 いざとなったら、Reason ファイルに生の JavaScript スニペットを貼り付けてください。
  • 素晴らしいエコシステム & ツール群。 お気に入りのエディタ、お気に入りの NPM パッケージ、そして、お気に入りの既存のスタックのあれやこれやそれのいずれかを使ってください。

* 信じられませんか？ JS - > Reason のチートシートをチェックするか、プレイグラウンド の Reason スニペットをいくつか試して、右側のアウトプットを観察してください。

なぜ後見言語として Ocaml なのか？なぜ [私のお気に入りの言語名が入る] ではないのか？

まず、私たちがどの言語を選んでも、皆さんからこの質問を受けることを理解してください！ =）

多くの後見言語は、前のセクションで挙げた点を満たします。しかし、下記の点は、私たちの検討の中で採用を断念している点です。

  • ネイティブコードにレンダリングする機能。 OCaml のネイティブ（アセンブリ）の起動時間は、1桁ミリ秒です。 私たちは、いつの日か Reason をネイティブに使用できるようにするという大きな計画があります。それまでの間は、優れた JavaScript 互換性に焦点を合わせます。
  • 副作用、ミューテーション & 他の非常口。 これらは通常、言語の輝かしいセールスポイントではありません。しかし、緻密な相互運用/書き換えなしに、コードベース一部に向けて橋渡しできることが Facebook の私たちにとって重要です。 OCaml のデフォルトはイミュータブルで関数的なコードですが、非常口を持つことで、しばしば初期導入を単純にできます。
  • 実装の問題を解決する。 OCaml は、20年に渡って洗練され、毎年良くなっています。 仮に新たなシンタックス & ツールチェインを提案したとして、コードベースの記述に80%の労力を費やした上に、破綻したセマンティクスで「わーい」とタイプするようなことは望みません。
  • React を書くための言語。 Reason の作者は ReactJS も作成しました。最初のプロトタイプはOCaml の遠い親戚である SML で書かれていました。 私たちは ReactML を広く普及させるために ReactJS に書き換えました。 数年経った今、私たちは ReasonReact を通して ReactJS の先に再び立っています。
  • 素晴らしい、成長するコミュニティ。 私たちは本当に素晴らしいと思っています。 私はカナダ人です。 私たちには世界中のメンバーがいます。 いざとなったら、Discord チャンネルで尋ねてください。あなたのタイムゾーン内のメンバーの誰かが答えてくれるでしょう。

Reason が気に入らない？

何らかの誤認/誤解を解消するために、まず Discord で話し合いをさせてください！ しかし、それでも Reason のことをまだ気に入らないなら、ここにいくつか代替を挙げます:

  • OCaml。Reason は OCaml 上の単なるシンタックスかつツールチェインのレイヤーです。OCaml<->Reason との相互運用では、同じ抽象構文木を共有するのでシームレスでなければなりません。BuckleScript は OCaml で自然にうまく動きます。Js_of_ocaml でも同様です。
  • Rust。Reasonの親しいいとこ分です。ガベージコレクションのない、スピードと安全性に注視した言語です。
  • Elm。Reasonのもうひとりのいとこです。ウェブアプリケーション開発をきれいで楽しいものにしてくれます。
  • PureScript。Haskell から着想を得て、ウェブにコンパイルされます。
  • Fable。OCaml に密接に関係している F# に基づいています。
  • ClojureScript。表面上は共通点が何もないように見えますが、こちらも簡潔さ & すぐれた相互運用性に重きを置きます。
  • Swift。「フルーツ」風味のOcamlです。
  • Haxe。基本的にはどの言語にもコンパイルできます。

助けが欲しい！もっと知りたい？コミュニティで私たちと会話しましょう！
