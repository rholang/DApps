Null, Undefined & Option

Reason 本身並沒有 null 或 undefined 的概念。 這是件很棒的事，因為它排除了所有這類型的錯誤。 再也沒有 undefined is not a function 和 cannot access foo of undefined 這種錯誤了！

然而，關於這種「可能不存在的值」的觀念仍然是有用的，它仍存在於我們的程式語言中。

我們將這種可能存在或不存在的值包成一個 option 型別。以下是它在標準函式庫裡面的定義：

type option('a) = None | Some('a)

這表示「型別為 option 的值會是 None (沒東西) 或是以 Some 包住的真正的值」。你可能會覺得為什麼我們不能把 Some 拿掉然後直接做下面的操作：

type option('a) = None | 'a

如此一來這個值會是 None 或是 真正的值。這樣雖然很方便，但有很多我們不希望這樣做原因。我們先在這裡省去解釋。

範例

以下是一個一般的值：

let licenseNumber = 5

為了呈現可能為 null 的概念，你會把它包成一個 option 的型別。為了更清楚的表達這個範例，我們用條件式來表示它：

let licenseNumber =
  if (personHasACar) {
    Some(5);
  } else {
    None;
  };

稍後，當另一段程式碼收到這樣的值時，它將被迫處理這兩種情況：

switch (licenseNumber) {
| None => print_endline("The person doesn't have a car")
| Some(number) =>
  print_endline("The person's license number is " ++ string_of_int(number))
};

透過將普通的數字轉換為 option 型別，並強制您處理 None 的情形，程式語言有效地消除了你處理不當或忘記處理一個概念上的 null 的可能性！

用法

Option 的輔助模組在這裡。

與 JavaScript 的 undefined 和 null 的互通

Option 型別非常常見，我們在編譯到 JavaScript 時可以對其進行特殊處理：

Some(5)

會直接被編譯成 5 ，而

None

會被編譯成 undefined！ 如果你知道有個在 JavaScript 的字串可能會是 undefined，將它包成 option(string)，你就完成了！ 相同地，你可以將 Some(5) 或是 None 送到 JS 端並期待它會被正確的解譯 =)

警告 1

「Option 轉換成 undefined」並不是完美的，因為在這邊 option 的值可以被組合起來：

Some(Some(Some(5)))

這仍然會編譯成 5，但以下變得很麻煩：

Some(None)

這被編譯為以下的 js:

Js_primitive.some(undefined);

Js_primitive 是什麼呢？ 為什麼它不能被編譯成 undefined？ Long story short, when dealing with a polymorphic option type (aka option('a), for any 'a), many operations become tricky if we don't mark the value with some special annotation. 如果這聽起來不合理，不要擔心；只需記住以下規則：

  • 千萬不要傳入嵌套的 option 值（例如 Some(Some(Some(5)))）到 JS 端。
  • 千萬不要標記從 JS 端來的值為 option('a)。一定要給出具體、非多態的型別。

警告 2

不幸的是，很多時候你的 JavaScript 值可能同時是 null或 undefined。 在這種情況下，不幸的是，您不能輸入這樣的值，例如: option(int)，因為我們的 option 類型在處理 None 時只檢查 undefined 而不是 null。

解決方案：更精細的 undefined & null Interop

為了解決這個問題，我們透過 Js.Nullable 模組提供了更詳細的 null 和 undefined 輔助工具。 這在某種程度上類似于 option 型別，但與它不同。

範例

若要創建 JS 的 null，請使用 Js.Nullable.null。 To create a JS undefined, use Js.Nullable.undefined (you can naturally use None too, but that's not the point here; the Js.Nullable.* helpers wouldn't work with it).

If you're receiving, for example, a JS string that can be null and undefined, type it as:

[@bs.module "MyConstant"] external myId: Js.Nullable.t(string) = "myId"

To create such a nullable string from our side (presumably to pass it to the JS side, for interop purpose), do:

[@bs.module "MyIdValidator"] external validate: Js.Nullable.t(string) => bool = "validate";
let personId: Js.Nullable.t(string) = Js.Nullable.return("abc123");

let result = validate(personId);

The return part "wraps" a string into a nullable string, to make the type system understand and track the fact that, as you pass this value around, it's not just a string, but a string that can be null or undefined.

option 的轉換

Js.Nullable.fromOption 可以從 option 轉換到 Js.Nullable.t。Js.Nullable.toOption 則反之。
