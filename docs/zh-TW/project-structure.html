Project Structure

這些是既有、未編纂的社群實踐，非正式的認同傳播。 可能在某個時間點會刪除某些東西，或者強迫其他。 目前，建議給新人安心。

檔案命名風格

檔案名稱的第一個字母大寫（又稱支首字母大寫風格）。

理由：模組名稱只能字母大寫。 新人常問如何對應檔案到模組，為什麼 draw.re 對應 Draw 模組，有時候使用第一個字母非大寫的識別子來引用模組。 使用 Draw.re 讓對應更加率直。 這也幫助某些彆扭的第一個字母非大寫檔名：uRI.re。

忽略 .merlin 檔案

由建置系統產生（至少 JS 工作流是這樣），你不需要手動修改它。不要將它納入版本控制。

理由：.merlin 供 Merlin 用來了解專案配置及編輯器的工具。 檔案內容包含絕對路徑，並非跨平台相容（例如，Windows 路徑並不相同）。

資料夾

Try not to have too many nested folders. Keep your project flat, and have fewer files (reminder: you can use nested modules).

Justification: The file system is a tree, but your code's dependencies are a graph. Because of that, any file & folder organization is usually imperfect. While it's still valuable to group related files together in a folder, the time wasted debating & getting decision paralysis over these far outweight their benefits. We'll always recommend you to Get Work Done instead of debating about these issues.

第三方相依套件

盡量降到最低。

理由：編譯、靜態的型別語言無法像動態的語言那樣輕鬆地對其相依性塑模，尤其當我們仍依賴 NPM/Yarn（中期為了降低學習曲線），這兩種都不是以 Reason/BuckleScript 為中心製作。 保持相依簡單 & 瘦小對減少衝突可能性有幫助（例如，兩個勢均力敵的相依或者介面碰撞）。

將第三方程式碼放在 dependencies，而不是 peerDependencies。

理由：你應該要精神上將它們視為實作細節，某天將會換掉。

文件

擁有它。 花點時間讓文件更好（範例、陷阱）及專業，而非只是花俏的外觀。 請使用範例，避免像是 foo 和 bar 的命名。 一直都會有更具體的命名（這只是個範例，並不需要抽象/概論）。 API 將會大量執行）。 對於部落格文章，請不要重複文件內容，請重新從舊到新的轉述並說明為什麼（例如，"這本來是元件，現在是函式，是因為..."）。

理由：對新人來說很難區分簡單/正派的函式庫還是只有漂亮的外觀。 為了社群的好，請不要太用力的搶先其他函式庫。 請廣傳，但也請用你自己的判斷力。

PPX & 其他 Meta-工具

盡量降到最低。 PPX，除非是在盛名情況下使用（printer、accessors 和 serializer/deserializer 生成），對新人造成非常大的學習浪花；建構在已經學習的語法、語意、型別、建置工具 & FFI 之上，學習函式庫個別自訂程式碼轉換是個額外的步驟。 更侵入的 macros 也會降低程式碼本身的語意意義，因為本質被藏在其他地方。

範式

不要過度濫用花俏功能。給未來 API 一些空間但不要過度設計。

理由：簡單的程式碼幫助新人了解及有潛力貢獻你的程式碼。 付出貢獻是最佳的學習方式。 你所得到的額外幫物可能也會超過使用一些更巧妙的語言技巧。 但是請在比較休閒的專案中使用新的語言技巧！ 你可能會發掘出新的程式碼設計。

發佈

如果是對 JS 函式庫的綁定，請不要發佈 JS 的人工製品。 如果是正統的函式庫，發佈 lib/js 裡面的人工製品，如果你覺得 JS 用戶可能會需要。 尤其當你是逐漸轉換 JS 函式庫到 Reason + BuckleScript，不會搞壞既有 JS 用戶。

請將"reason" 和 "bucklescript" 關鍵字放入 package.json 的 keywords 欄位中。未來讓我們更容易找到這些繫結。

理由：請對函式庫的 JS 用戶好一些，他們是未來的 Reasoners。
