Variant!

看，Reason 資料結構的皇冠寶石！

多數語言的多數資料結構是關於 "這個以及那個"。變異允許表達成 "這個或者那個"。

type myResponseVariant =
  | Yes
  | No
  | PrettyMuch;

let areYouCrushingIt = Yes;

Yes、No 及 PrettyMuch 不是字串，不是參照，也不是某些特殊資料型別。 這些稱之為 "建構子（constructors）"（或者 "tag"）。 | 棒子符號分隔每個建構子。

注意：變異的建構子必須是大寫開頭。

用法

Reason 中跟變體一起的一個重要功能，switch 表達式。

Reason 的 switch 看起來很類似其他語言的 switch（意即大塊的 if/elseif/elseif...）。 允許你檢查每一個可能的變異。 若要使用它，列舉出你想要使用的每一個變異建構子，後面接著 =>，再來就是該變異案例下的表達式。

let message =
  switch (areYouCrushingIt) {
  | No => "No worries. Keep going!"
  | Yes => "Great!"
  | PrettyMuch => "Nice!"
  };
/* message 為 "Great!" */

變異具有極其豐富的型別系統協助。 例如，若是忘了包含任一個案例，會發生型別錯誤，或者，存在多餘的相同案例。 請務必看看後續章節的 switch 及樣式匹配！

變異必須明確定義

若使用不同檔案中的變異，將它帶入範疇，像是 record 的作法：

/* Zoo.re */

type animal = Dog | Cat | Bird;

/* example.re */

let pet: Zoo.animal = Dog; /* 首選 */
/* 或者 */
let pet = Zoo.Dog;

建構子參數

變異建構子可以持有額外資料，使用逗點分隔。

type account =
  | None
  | Instagram(string)
  | Facebook(string, int);

這裡，Instagram 持有 string、Facebook 持有 string 及 int。用法：

let myAccount = Facebook("Josh", 26);
let friendAccount = Instagram("Jenny");

有注意到建構子用法就像是呼叫函式嗎？ 就好像 Facebook 是個函式，接受兩個參數。 這並不是個巧合；這就是為什麼建構子的資料稱為 "建構子參數" 的原因。

使用 switch，可以樣式匹配（一樣，稍後章節討論）建構子的參數：

let greeting =
  switch (myAccount) {
  | None => "Hi!"
  | Facebook(name, age) => "Hi " ++ name ++ ", you're " ++ string_of_int(age) ++ "-year-old."
  | Instagram(name) => "Hello " ++ name ++ "!"
  };

特別表揚

標準函式庫 公開兩個重要的變異你將會經常聽到。

option

type option('a) = None | Some('a);

這是個慣例，用來模擬其他語言的 "空值"（又稱 undefined 或者 null）。 感謝這個方便的型別定義，Reason 可以預設所有的值都不會是空值。 int 永遠都是 int，不會 "int 或者 null 或者 undefined"。 如果要表達 "可空值的 int"，就使用 option(int)，可能有的值就是 None 或者 Some(int)。 switch 強迫你處理這兩個情況；因此，純 Reason 程式不存在 null 錯誤。

list

type list('a) = Empty | Head('a, list('a));

並不是真的型別定義。只是個例子。

表示： "某個 list 持有 a 型別的值（無論什麼型別），這個 list 可能是空的，或者是持有某個值加上額外的 list"。

Reason 給予 list 一個語法糖。 [1, 2, 3] 概念上跟 Head(1, Head(2, Head(3, Empty))) 相等。 再一次，switch 強迫你處理此變異的所有情況，包含 Empty（又稱 []）。 這消除另一個大的存取 category 的臭蟲。

其他類似變異的型別

你知道你可以在 string、int、float、array 以及其他大多數的資料結構上使用 switch 嗎？試試看！

提示 & 技巧

請小心不要將「帶有 1 個 tuple 參數的建構子」與「帶有 2 個參數的建構子」搞混：

type account =
  | Facebook(string, int) /* 2 個參數 */;
type account2 =
  | Instagram((string, int)) /* 1 個參數 - 是個 2-tuple */;

變異必須有建構子

如果你來自無須型別的語言，可能會嘗試 type foo = int | string。 Reason 是不可能的；必須給予每個分支一個建構子： type foo = Int(int) | String(string)。 這種情況通常是個 anti-pattern。 下列「設計決策」會有更詳細解釋。

與 JavaScript 交互使用

此段內容假設已有 BuckleScript 的 FFI 相關了解。 如果還不想應用變異於綁定 JS 函式，請先跳過。

許多 JS 函式庫的函式接受多種不同型別的參數。 這些情況下，十分建議使用變異。 例如，假設有個 myLibrary.draw JS 函式，接受 number 或者 string 參數。 你可能會試著綁定：

/*  內部保留使用 */
[@bs.module "myLibrary"] external draw : 'a => unit = "draw";

type animal =
  | MyFloat(float)
  | MyString(string);

let betterDraw = (animal) =>
  switch (animal) {
  | MyFloat(f) => draw(f)
  | MyString(s) => draw(s)
  };

當然可以這麼做，但有更好的方法！例如，簡單使用兩個 external 編譯同個 JS 呼叫：

[@bs.module "myLibrary"] external drawFloat : float => unit = "draw";
[@bs.module "myLibrary"] external drawString : string => unit = "draw";

BuckleScript 也提供了幾個方式供使用。

變異型別是從欄位找出

請參閱 record 章節。 變異也是一樣：函式不接受同時有兩個變異具有相同建構子。 同樣，存在這樣的功能，稱之為多形變異（polymorphic variant）。 以後再來談這個 =)。

設計決策

變異，是像是 Reason 這類型別系統的功能，有多種形式（多形變異、開放變異、GADT 等）。 前述提到的 option 變異，舉例來說，消滅對可空值型別的需求，其他語言的主要臭蟲來源。 哲學上來說，一個問題是由多個可能分支/條件組成的。 錯誤的處理這些條件是大多數我們所稱的臭蟲。 型別系統並不會神奇地消除這些臭蟲；而是指出未處理的條件並要求你處理它*。 具有正確地塑模 "這個 或 那個" 的能力十分重要。

例如，某些人對於型別系統安全地消除格式錯誤的 JSON 傳播到程式感到驚訝。 不會，不是透過系統！ 但是，如果剖析器傳回 option 型別 None | Some(actualData)，接著在呼叫方你必須明確地處理 None 情況。 就是這樣。

性能面，變異潛在地極大地加快你的程式邏輯。下列是段 JavaScript 片段：

let data = 'dog';
if (data === 'dog') {
  ...
} else if (data === 'cat') {
  ...
} else if (data === 'bird') {
  ...
}

這是線性（O(n)）的分支量檢查。跟 Reason 變異的比較：

type animal = Dog | Cat | Bird;
let data = Dog;
switch (data) {
| Dog => ...
| Cat => ...
| Bird => ...
}

編譯器看到變異，接著

  1. 概念地轉換成 type animal = 0 | 1 | 2
  2. 編譯 switch 到一個常數時間格式（O(1)）。

你可能想知道為什麼型別的函數式語言經常用來剖析；效率地 switching 龐大樹狀且安全，幾乎是變異的最佳使用情境。

腦袋清醒了嗎？ 變異與其他數學領域有深層連結； 請見這裡有趣的解釋。

* 與其讓設計遠離問題不如訴諸型別系統來涵蓋陷阱；事實上，處理每一個問題是不切實際，或者，單單為了設計一個解決方法而去完全了解每一個問題。 型別系統讓你安全地對程式碼做較大的改變，不需要先了解整個系統。 這對引導探索很好。 在這方面，型別允許我們不需要過度地設計 API 只為了繞過呼叫端的簡單陷阱。 減少了讓 "完成事情" 的抽象層面，減少呼叫端的認知負荷。
