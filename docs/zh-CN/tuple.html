Tuple

元组特性：

  • 不可变的
  • 有序的
  • 创建后长度固定
  • 组成类型不固定（能够存放不同类型的值）
let ageAndName = (24, "Lil' Reason");
let my3dCoordinates = (20.0, 30.5, 100.0);

元组的类型也可以在类型注释中使用。元组类型视觉上类似于元组值。

let ageAndName: (int, string) = (24, "Lil' Reason");
/* a tuple type alias */
type coord3d = (float, float, float);
let my3dCoordinates: coord3d = (20.0, 30.5, 100.0);

** 注意 **：没有大小为1的元组。你只需要使用这个值本身。

Usage

标准库提供 fst 和 snd函数 ( 查看此处 , 在 "Pair operations"下面)，使你能够访问2元组的第一个和第二个元素。 通常, 你可以通过解构方法访问元组的第n个成员 (稍后在边栏中介绍)：

let (_, y, _) = my3dCoordinates; /* now you've retrieved y */

_ 表示你忽略了元组的该项成员。

元组不可以用更新来改变，你可以通过解构旧的元组来创造新的元组。

提示& 技巧

你可以在使用元组方便的情况下, 通过元组传递你需要的多个值。例如，当函数返回值时多个时：

let getCenterCoordinates = () => {
  let x = doSomeOperationsHere();
  let y = doSomeMoreOperationsHere();
  (x, y)
};

或与 "模式匹配" (后面章节会讲) 结合使用：

switch (isWindowOpen, isDoorOpen) { /* this is a 2-tuple */
| (true, true) => ...
| (true, false) => ...
| (false, true) => ...
| (false, false) => ...
}

请保持元组的 ** 局部 ** 用法。对于长期生存并经常传递的数据结构, 首选 具有命名字段的 ** 记录 **。

元组类型也可以称为 "product type", (string、int) 在某些位置写作 string * int。 这个想法是 tuple 实际上是一个 "cartesian product"; 设想一个2D 网格, 在 x 轴上有 string, y 轴上 int!

Tuple + switch 的组合非常强大且简洁, 清除了整个类别的 bug。 组合起来, 它们清晰地列出了所有可能的值的组合. 一个 (bool, bool) 的 tuple 明确的含有 2 * 2 = 4 种可能性, 类型系统会要求你覆盖全部 4 种. This lends itself well to code refactors; instead of informally adding a few if-else on arbitrary values here and there, you can directly pinpoint the exact switch branch you need to alter, no more and no less.

设计决策

Tuple 的存在对于那些无类型语言来说可能看起来很奇怪. " 为什么不只用 list/array? "

A type system isn't all-powerful, nor should it be; some tasteful trade-offs need to be applied in order to keep the language simple, performant (both compilation and running speed) and easy to understand. Reason 的 list, 例如, 它们是尺寸灵活的; 它们可以被 串联, 追加, 裁剪 等等. In return, they need to be homogenous (can only contain a single type of value per list), and random index access on them might not always be valid*. Tuple, on the other hand, through its constraint on size, is faster, gives the type system the leeway to exhaustively track all its items' types, and guarantees safe access. In general, you'll notice a few prominent, tasteful tradeoffs in a type system: record fields are fixed but can be heterogenous, while a map's fields are flexible but homogenous, etc.

A Reason tuple is typed "structurally". This means that even if you don't annotate your data with an explicit type, the compiler can still deduce it by looking at its content, its usage, etc. As long as the declarations and the usages' inferred shapes match up, you're all good!

* It's not that the Reason type system cannot accept heterogenous, dynamically-sized lists; it actually can (hint: GADT)! But making such feature the default increases both the first-time learning overhead and the understandability of code. Just because the types can accomplish it doesn't mean it's always a good idea to let some pieces of code grow unboundedly complex!
