JSX

想在 Reason 里写 HTML 语法吗？如果不想，跳过这个章节，假装什么都没看见！

Reason 支持 JSX 语法，和 ReactJS 相比存在细微的差异。 Reason JSX 并没有和 ReactJS 绑定；最终会转换成一般的函数调用：

ReasonReact 读者请 注意：这不是 ReasonReact 转换 JSX。详情见用法章节。

大写首字母标签

<MyComponent foo={bar} />

变成

([@JSX] MyComponent.createElement(~foo=bar, ~children=[], ()));

非大写首字母标签

<div foo={bar}> child1 child2 </div>;

变成

([@JSX] div(~foo=bar, ~children=[child1, child2], ()));

Fragment

<> child1 child2 </>;

变成

([@JSX] [child1, child2]);

Children

<MyComponent> foo bar </MyComponent>

这个语法是为了传递一个包含两个元素的列表，foot 和 bar，至子节点的位置。去除语法糖之后，实际上是一个列表，里面包含 foot 和 bar：

([@JSX] MyComponent.createElement(~children=[foo, bar], ()));

再次 注意 这不是 ReasonReact 的转换结果；ReasonReact 会将最终列表变成数组。但意思是差不多的。

因此，自然地，<MyComponent> foo </MyComponent> 去除语法糖后则是 ([@JSX] MyComponent.createElement (~children=[foo], ()))。 也就是说, 无论做什么, 传递给子级位置的参数都将被包装在列表中。 如果你并不想这样？ 如果你想直接传递 foo 而不被其他元素包裹？

Children 展开

为了解决上面的问题，我们引入了

<MyComponent> ...foo </MyComponent>

这会传递 foot 而不是 包在一个列表中（或者是数组，ReasonReact）。也就是，会去语法糖至：

([@JSX] MyComponent.createElement(~children=foo, ()));

这对于 foot 已经是一个列表，而且想直接使用而不是再包裹一层的情况（这回出现一个类型错误），会非常有用。 它还允许您在 children 位置传递任意数据结构（请记住， JSX 的children 真的只是一个普通的 prop）：

<MyComponent> ...((theClassName) => <div className=theClassName />) </MyComponent>;
<MyForm> ...("Hello", "Submit") </MyForm>;

用法

有关 JSX 的示例代码，请参阅 ReasonReact，将上述调用转换为特定于 ReasonReact 的调用。

下面这段代码展示了 JSX 大多数的功能。

<MyComponent
  booleanAttribute={true}
  stringAttribute="string"
  intAttribute=1
  forcedOptional=?{Some("hello")}
  onClick={send(handleClick)}>
  <div> {ReasonReact.stringToElement("hello")} </div>
</MyComponent>

区别于 JS JSX

  • 属性和 children 不要求 {}，但无论如何，我们还是会为学习方便而显示它们。一旦 refmt 你的文件，其中一些会消失，一些会变成括号。
  • 不支持 JSX 属性展开: <Foo {...bar} />。虽然有些相关，但我们确实有 children 展开语法，像上面提到的：<Foo> ...baz </Foo>
  • 双关!

双关

“双关”是指标签和值相同时的语法缩写。 就像在 JavaScript 中，可以 return {name} 而不必 return {name: name}。

Reason JSX 支持双关。 <input checked /> 只是 <input checked=checked /> 的缩写。 formatter 程序将帮助你在可能的情况下，format 到后一种写法。 这在需要传递很多属性的时候，会很方便。

<MyComponent isLoading text onClick />

因此，在用额外的库来解决过多的属性传递问题之前，Reason JSX 组件可以塞进更多的属性。

注意，这也是和 ReactJS JSX 的区别所在，因为后者 没有 双关语法。 ReactJS 的 <input checked /> 其实是 <input checked=true /> 的语法糖，为了更符合 DOM 的习惯以及向后兼容性。

提示& 技巧

对于希望使用 JSX 库作者：上面的 [@JSX] 属性内在是 ppx 宏的 hook，用于标记希望格式化为 JSX 的函数。 一旦你标记了函数，就可以把它转换成任意表达式。

这样一来，每个人都可以受益于 JSX 语法，而无需选择使用它的特定库，例如 ReasonReact。

JSX 调用支持 标记函数：可选、显式传递可选以及有默认值的可选。

设计决策

这样设计 JSX，是基于我们对于帮助语言进化的考量。 可以看看 这篇博文中 "What's the point?" 章节 。

语言层面包含宏指令 + 库对于 JSX 不可知的能力，使得每个库都能潜在地利用 JSX 而不费太大力气。 这样，我们为底层的 OCaml 语言增加了一些使用上的熟悉度，而不影响它的语义（也就是它的执行方式）。 Reason 的一个重要目标，是让更多的人能利用上 OCaml 这种美妙的语言，同时放弃围绕语法和格式化的耗时争论。

* 你可能想知道为什么你从来不需要在 ReactJS 使用 children 展开；ReactJS 使用一些特殊的运行时逻辑 + 特殊的语法转换器 + 可变参数检测 & 使用标记避免大部分 case（看这里）。 这种动态的用法会让类型系统的检测复杂 很多。 由于控制了整个语法和 ReasonReact，所以我们决定引入 children 展开以消除元素包裹与不包裹的情况之间的歧义，而无需增加编译时 & 运行时成本！
