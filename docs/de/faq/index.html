Frequently Asked Questions

Sollte ich zuerst Reason oder OCaml lernen?

Es ist nicht nötig, zu wählen! Reason und OCaml teilen exakt die gleiche Semantik (d.h. wie der Code ausgeführt wird). Nur die Syntax unterscheidet sich. Nutze Reason-tools, sodass du frei zwischen den beiden Syntaxen konvertieren kannst. Ein Reason-Tutorial ist ein OCaml-Tutorial und umgekehrt. Im Terminal könntest du folgende Aliase verwenden:

# konvertiert ocaml code zu reason
alias mlre="pbpaste | refmt --parse ml --print re --interface false | pbcopy"
# konvertiert reason code zu ocaml
alias reml="pbpaste | refmt --parse re --print ml --interface false | pbcopy"

Sie nehmen deinen Code aus der Zwischenablage (macOS), konvertieren ihn und fügen ihn wieder in deine Zwischenablage ein! Tausche pbpaste / pbcopy mit den Zwischenablagefunktionen deines Systems aus.

Ich bin mir nicht sicher, was ich mit Reason tun soll

Think of what project you'd usually make if it was pure JavaScript; try porting/writing that in Reason + BuckleScript instead! We recommend trying to make concrete, end-user projects (e.g. a little command line util) rather than infra-level projects (e.g. a boilerplate generator). The latter category requires expertise and understanding idiomatic Reason code.

Wie hängen Reason, BuckleScript und OCaml zusammen?

Reason's a syntax for OCaml and supports all its features. BuckleScript compiles OCaml/Reason code into JavaScript.

Wo kommen all diese print_endline, string_of_int Funktionen her?

Sie stammen aus der Standardbibliothek und sind bereits während der Kompilierung deiner Datei geöffnet. Daher liegen sie im Sichtbarkeitsbereich.

Du kannst mehr über die Pervasives Bibliothek in der API Dokumentation erfahren:

https://reasonml.github.io/api/Pervasives.html

Kann ich eine Funktion zur Ausgabe beliebiger Datenstrukturen haben?

Wenn du mit BuckleScript nach JavaScript kompilierst, kannst du JS console.log durch Js.log verwenden. Wenn du zu nativem Code kompilierst, dann kannst du ppx_show verwenden. Ein zukünftiges OCaml-Feature (modular implizit genannt) löst das direkt in der Sprache.

Warum gibt es ein +, um ints zu addieren, und +. für die Addition von floats usw.?

Siehe hier.

Funktioniert die Bibliothek ___ mit Reason?

Die meisten JS-Bibliotheken sollten ganz einfach unter Reason + BuckleScript funktionieren. Auf der nativen Seite, da Reason nur eine Syntax-Transformation ist: Ja, sie arbeiten auch mit Reason. Aber der native Workflow ist zur Zeit in Arbeit und muss noch ausgefeilt werden.

Wie sieht es serverseitige aus? Sollte ich zu nativem Code oder zu JS kompilieren und node.js benutzen?

Wir kompilieren zu nativem Code, aber der native Workflow ist derzeit in Arbeit. Zu diesem Zeitpunkt empfehlen wir die Kompilierung nach JS über BuckleScript und die Verwendung der JS-Wrapper in reasonml-community oder anderswo.

Wie sieht es mit asynchronem Code in BuckleScript aus?

Erstens, wenn du keine Verbindung zu einer Bibliothek herstellst, die Promises verwendet, kannst du einfach Callbacks verwenden. Jeder versteht sie und sie sind leistungsfähig.

Wenn du eine Bindung an eine JS-Bibliothek herstellen musst, die Promises verwendet, oder mit einer solchen Bibliothek kommuniziert, kannst du Js.Promise von BS verwenden. Es gibt auch die Möglichkeit, in Zukunft etwas syntaktischen Zucker zu haben. Auf lange Sicht würden wir gerne eine spezifikationskonforme Promise-Implementierung in OCaml / Reason richtig umsetzen, so dass die Compiler-Optimierungen greifen könnten.

Für eine idiomatischere OCaml-Lösung: auf der nativen OCaml-Seite haben wir lwt und Async. Wir verwenden sie aktuell nicht im Web, aber wir könnten dies in der Zukunft tun.

Wie sieht es mit (Unit) Tests aus?

Einige der OCaml-Sprachfunktionen (nicht nur Typen) können möglicherweise die Notwendigkeit für Unit-Tests auf später verschieben. In der Zwischenzeit, um nach JS zu kompilieren, arbeiten wir an Jest wrapper. Wir werden uns auch mit Jest für nativ beschäftigen, wenn Jest in Zukunft mit Reason geschrieben wird (noch kein konkreter Plan). OUnit ist aktuell eine gute, kleine native OCaml Testing-Bibliothek.

Was ist die .merlin-Datei im Wurzelverzeichnis meines Projekts?

That's the metadata file for editor support. This is usually generated for you; You don't need to check that into your version control and don't have to manually modify it.

Ich sehe keinen import oder require in meiner Datei. Wie funktioniert die Modulauflösung?

Bei Reason / OCaml musst du keinen Import schreiben. Module, auf die in der Datei Bezug genommen wird, werden automatisch im Projekt gesucht. Insbesondere fordert ein Modul Hello den Compiler auf, nach der Datei hello.re oder hello.ml zu suchen (und ihrer entsprechenden Interface-Datei, hello.rei oder hello.mli, falls vorhanden).

Ein Modulname ist der Dateiname, groß geschrieben. Es muss pro Projekt einzigartig sein. Dies abstrahiert das Dateisystem und ermöglicht es dir, Dateien zu verschieben, ohne den Code zu ändern.

Sind Some | None, contents, Array, List und all diese speziell? Wo kommen die her?

Sie sind gewöhnliche Varianten/Records/Moduldefinitionen, die mit der Standartbibliothek kommen und standardmäßig aus Gründen der Bequemlichkeit während der Kompilierung geöffnet sind (open).

Was bedeutet ein Argument mit einem vorangestellten Unterstrich (z. B. _ oder _foo)?

Angenommen du hast List.map(item => 1, myList);. Das Argument item wird nicht verwendet und wird eine Compiler-Warnung ergeben. Die Verwendung von _ => 1 zeigt stattdessen an, dass du das Argument absichtlich erwartest und ignorierst, womit du die Warnung umgehst. Alternativ hat _item => 1 die gleiche Auswirkung aber es deutet mehr beschreibend an, was du ignorierst.

Was ist dieses MyModule.t, dass ich ständig sehe?

Angenommen, MyModule ist der Name eines Moduls, t ist eine Community-Konvention, die „den Typ angibt, der dieses Modul repräsentiert, was auch immer das bedeutet“. Zum Beispiel, für das Js.String-Modul ist String.t der Typ, der herumgereicht wird und „einen String“ repräsentiert.

Warum gibt es ein Js_promise und auch ein Js.Promise? Was ist mit Js_array, Js_string und was noch?

Als Konvention ist Js_foo das eigentliche Modul, und Js.Foo ist nur ein Alias dafür. Sie sind äquivalent. Gebe Js.Foo den Vorzug, denn das ist der offizielle, öffentliche Modulname.

Wann werden modulare implizite & multicor & algebraische Effekte fertig sein?

Werden sie eines Tages. In der Zwischenzeit, hilf uns mehr Reason-Code auszuliefern! Die Popularität wird helfen, mehr OCaml-Beiträge zu ergeben. Je weniger sich die OCaml-Leute um einfache Aufgaben kümmern müssen, desto mehr können sie sich auf großartige Forschung und Umsetzung konzentrieren!

Warum sind BuckleScript und bsb so schnell? Wie kann ich sie abbremsen?

BuckleScript ist für die Performance über den gesamten Stack optimiert. Du kannst versuchen, es zu verlangsamen, indem du ein Dutzend Ebenen von Umleitungen und Metaprogrammierung hinzufügst. Probiere:

  • Ein paar Endlosschleifen hier und da einzufügen.
  • Ein JavaScript-Build-Werkzeug in die Pipeline reinzustopfen.
  • Mehr Abhängigkeiten für ein Hello World reinzuziehen.

Ich sehe, dass eine eigenartige .cmi/.cmx/.cmj/.cma Datei in einem Compiler-Fehler referenziert wird. Wo kommen diese Dateien her?

Die OCaml-Community verwendet häufig Dateierweiterungen, um je nach Build-Ziel (nativ / Bytecode / JavaScript) zwischen Quellentypen, Artefakten und Metadaten zu unterscheiden. Im Folgenden findest du eine Übersicht über einige Dateierweiterungen, auf die du stoßen könntest:

Quelldateien

  • .ml: OCaml Quelldatei
  • .mli: OCaml Interface-Datei; legt fest, welche Teile der passenden ml-Datei für die Außenwelt sichtbar sind
  • .re: Reason-Quelldatei. Wie .ml, aber für Reason
  • .rei: Grund Interface-Datei. Wie .mli, aber für Reason

Kompilierte Dateien

  • .cmi: Kompilierte Interface-Datei (.rei/mli)
  • .cmx: Kompilierte Objektdatei für native Ausgabe (via ocamlopt)
  • .cmo: Kompilierte Objektdatei für Bytecode-Ausgabe
  • .cmj: Kompilierte Objektdatei für Web (via BuckleScript)
  • .cma: Bibliotheksdatei für Bytecode-Ausgabe (äquivalent zu .a-Dateien in C)
  • .cmxa: Bibliotheksdatei für native Ausgabe
  • .cmt: Enthält einen „Typedtree“ – grundsätzlich der AST mit der ganzen Info zu Typen
  • .cmti: Entspricht einer .cmt-Datei aber für Interface-Dateien
  • .cmxs: Dynamisch geladenes Plugin (für native Kompilierung)
  • .o: Kompilierte native Objektdatei
  • .out: Konventioneller Name / Erweiterung für die endgültige Ausgabe, die von ocamlc / ocamlopt erzeugt wurde (z. B. ocamlc -o myExecutable.out)

Andere OCaml-Dateien im Ökosystem

  • .mll: Definitionsdatei für lexikalische Analyse mit ocamllex
  • .mly: Definitionsdatei für Parsergenerator mit ocamlyacc
  • .mldylib: Enthält eine Liste von Modulpfaden, die zusammen kompiliert und archiviert werden, um ein entsprechendes .cmxs-Ziel (natives Plugin) zu erstellen
  • .mliv: Datei spezifisch für Batteries für gewisse benutzerdefinerte Vorverarbeitungen.
  • .mllib: OCaml Bibliothek (cma und cmxa)
  • .mlpack: OCaml Paket (cmo erzeugt mit dem -pack Flag)
  • .mlpp: Extlib-spezifische Dateien für gewisse benutzerdefinerte Vorverarbeitungen
  • .mltop: OCamlbuild top-level Datei, verwendet von OCamlbuild, um eine .top-Datei zu generieren
  • .odocl: OCaml Dokumentationsdatei

Wenn einige dieser Erklärungen noch ein wenig kryptisch sind, hier sind Ausführungen zu einigen der oben verwendeten Begriffe: - AST: Abstract Syntax Tree. Die Datenstruktur, die aus dem Quellcode stammt, auf dem der Compiler operiert. - Linking: Der Schritt, in dem der Compiler viele kompilierte Zwischendateien behandelt und sie zusammensetzt. Z. B. wird A mit B verbunden, weil die Ausgangsquelldatei von A die Datei B referenziert. - Nativ: Builds, die auf Bare-Metal-Assembler-Anweisungen der betreffenden Plattform ausgeführt werden. - Bytecode: Wie nativer Code, aber portabler und weniger performant - Objektdatei: Enthält Maschinencode, der nicht direkt ausführbar ist.

Für viele dieser Dateierweiterungen gibt es mehr Informationen und Kontext auf der OCaml-Seite und in diesem Mailinglistenbeitrag. Im OCaml-Handbuch gibt es tiefere Ausführungen über native und Bytecode-Kompilierung, die detailliertere Beschreibungen enthalten.
