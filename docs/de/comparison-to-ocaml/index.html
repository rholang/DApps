Comparison to OCaml

Wenn du von OCaml kommst oder als ein Neueinsteiger ein Tutorial über OCaml liest, ist diese Anleitung für dich! Aber vergiss nicht, dass reason-tools spontan zwischen OCaml- und Reason-Syntax konvertieren können.

Kommentare

OCamlReason
(* OCaml (*nest*) *)/* Reason /*nest*/ */

Umbenennung von Operatoren

Reason hat alle Infix-Operatoren von OCaml, aber einige Operatoren werden anders ausgedrückt. In Reason wird strukturelle Gleichheit als == und referenzielle (physikalische) Gleichheit wird als === geschrieben. In Reason, um die entsprechende Ungleichheit zu erhalten, ersetze das erste Zeichen mit einem !-Zeichen. (!= für strukturelle Ungleichheit und !== für referenzielle Ungleichheit).

GleichheitOCamlReason
Strukturellx = yx == y
Referenzx == yx === y
UngleichheitOCamlReason
Strukturellx <> yx != y
Referenzx != yx !== y

Lokaler Sichtbarkeitsbereich

Die lexikalische Sichtbarkeit in Reason (lexical scope) ist genau die gleiche wie in OCaml, aber let-Bindungen ähneln auf syntaktische Weise „block scope“, der vielen Entwicklern vertrauter ist. In Reason werden sie mit {}-Klammern erstellt, welche sowohl let-Bindungen und imperative Anweisungen enthalten dürfen, getrennt durch ;. Alle Blöcke werden bis zur letzten Zeile ausgewertet und das Semikolon in der letzten Zeile ist optional. {}-Klammern werden nur benötigt wenn du mehr als ein Element zu mit ; zu verketten hast.

  Reason
</th>

<td>
  <pre><code>{

let msg = "Hello"; print_string(msg); let msg2 = "Goodbye"; print_string(msg2) };

OCaml 
let _ =
  let msg = "Hello" in
  print_string msg;
  let msg2 = "Goodbye" in
  print_string msg2

Reasons {}-Syntax entfernt viele allgemein berichtete Schmerzpunkte in OCamls Syntax:

  • Doppelte Semikolons werden vollständig entfernt.
  • begin/end werden vollständig entfernt.
  • Berüchtigte imperative Parsing-Probleme sind verschwunden.
  • Modulrümpfe und lokaler Sichtbarkeitsbereich wurden vereinheitlicht.

Lokaler Bereich vs. Modulrumpf

In Reason: alles, was zwischen {} in lokalen Sichtbarkeitsbereichen und in Modulrümpfen sein kann. Du kannst Code normalerweise zwischen den beiden Kontexten ausschneiden/einfügen. In OCaml sind die Syntaxen für die beiden Kontexte sehr unterschiedlich. Lokaler Sichtbarkeitsbereich erfordert ein abschließendes in, aber Modulrümpfe nicht und einige imperative Anweisungen müssen _ oder () zugewiesen werden, oder verwende double ;;.

Match-Funktionen mit einem Argument

OCaml hat eine Funktionsdefinition (function |), die als äquivalent zu function a -> match a with ... angesehen wird. Reason hat die gleiche aber die Syntax stellt klar heraus, dass sie tatsächlich eine Erweiterung einer Funktion mit einem Argument ist. Der Einzelfall-Match ist eine natürliche Erweiterung des einfachen Lambdas, und Lambda mit mehreren Fällen ist eine natürliche Erweiterung des Einzelfall-Lambdas.

Argumente annotieren

Sowohl in Reason als auch in OCaml werden Argumente mit Typen dadurch annotiert (wie bei allem anderen), dass sie von Klammern umgeben werden nachdem :typeAnnotation angehängt wurde.

(arg: argType) => returnValue;

(arg: argType) => (arg2: arg2Type) => returnValue;

(arg: argType, arg2: arg2Type) => returnValue;

Sowohl Reason als auch OCaml erlauben die Annotation des Rückgabetyps mit der Form der let-Bindung, die „besonders viel syntaktischen Zucker“ hat.

(* OCaml *)
let myFunc (a:int) (b:int) :int * int = (a, b)
let myFunc (a:int) (b:int) :int list = [1]
let myFunc (a:int) (b:int) :int -> int = fun x -> x + a + b

/* Reason */
let myFunc = (a: int, b: int) :(int, int) => (a, b);
let myFunc = (a: int, b: int) :list(int) => [1];
let myFunc = (a: int, b: int): (int => int) => (x) => x + a + b;

Da wir => für alle Funktionen in Reason verwenden, gibt es einen Fall, in dem wir zusätzliche Klammern um einen Rückgabetyp hinzufügen müssen, der selbst ein Funktionstyp ist.

Typparameter

OCaml

OCamls Typanwendungen (etwa wie „generics“) werden in umgekehrter Reihenfolge angewendet.

Mit OCaml führt dies zu einigen unintuitiven Konsequenzen.

let x: int list = [2]

type listOfListOfInts = int list list

(* Wird geparst als: *)
type listOfListOfInts = (int list) list

Die Dinge werden sogar noch seltsamer, wenn Typkonstruktoren mehrere Parameter akzeptieren. Mehrere Argumente erfordern Klammern und Kommas, um Typparameter voneinander zu trennen, aber diese Klammern stellen keine Tupel dar. Die Klammer/Komma-Form muss auch angegeben werden, wenn Typinstanzen wie z. B. (int, string)-Tupel konstruiert werden.

type ('a, 'b) tuple = 'a * 'b

type listOfTuplesOfStringAndInt = (string, int) tuple list

(* Was geparst wird als: *)
type listOfTuplesOfStringAndInt = ((string, int) tuple) list

(* Was eine Liste von (Tupel von (string und int)) erlaubt *)
let tuples: listOfTuplesOfStringAndInt = [("asdf", 3)]

Reason

Zusammengefasst vereinheitlicht Reason fast die ganze Syntax in einfachem „Funktionsaufruf“-Stil, was bedeutet, dass Typparameter dem gleichen Komma-separiertem Muster folgen, wie sonst überall in der Syntax zu sehen. Dies führt zu weniger syntaktischen Mustern, die zu lernen sind.

Zum Beispiel kannst du dir vorstellen, dass list eine „Funktion“ für Typen ist, die einen Typ akzeptiert und einen neuen Typ zurückgibt.

Tupel als Typparameter

Weil OCaml Klammern und Kommas verwendet, um mehrere Argumente in Typkonstruktoren zu repräsentieren, ist es verwirrend, wenn eins der Argumente eines Typkonstruktors selbst ein Tupel ist. In OCaml ist es schwierig, sich an den Unterschied zu erinnern zwischen einem Typkonstruktor mit mehreren Argumenten und einem Typkonstruktor mit einem einzelnen Argument, das ein Tupel ist.

Die folgenden Beispiele zeigen den Unterschied zwischen zwei Typparametern für pair und einem einzelnen Parameter, der ein Tupel ist.

OCamlReason
type intPair = (int, int) pairtype intPair = pair(int, int)
type pairList = (int * int) listtype pairList = list((int, int))
  • In Reason sieht die Syntax, die Tupel oder Tupeltypen repräsentiert, immer wie Tupel aus.
  • In Reason sieht die Syntax, die Records oder Record-Typen repräsentiert, immer wie Records aus.
  • So ziemlich alles andere verwendet das syntaktische Muster einer Funktionsanwendung (durch Kommas getrennte Argumente).

Varianten

OCaml
  • OCaml erwartet bereits, dass Konstruktorargumenttypen in Tupelform angegeben werden. Daher ist es verwirrend, wenn ein einzelner Konstruktor ein einzelnes Argument akzeptiert, das ein Tupeltyp ist.
  • Es ist sogar noch verwirrender, dass die Konstruktoren eigentlich keine Tupel akzeptieren und dennoch die Syntax Tupeln zu ähneln scheint.
  • Manchmal überschneiden sich die Syntax für die Instanziierung eines Konstruktors mit mehreren Argumenten und die Syntax für die Erzeugung einer Variante mit einem einzelnen Argument, das ein Tupel ist - daher sieht es genauso aus als wenn du einen Tupel übergibst, auch wenn du eigentlich keinen Tupel übergibst.
Reason
  • Variantenkonstruktortypen werden als durch Kommas getrennte Listen erwartet, wobei Klammern verwendet werden, um den Vorrang zu gruppieren (so wie bei allem anderem).
  • Instanzen der Variante zu konstruieren (wie du wohl erraten hast) folgt dem Funktionsanwendungsstil (durch Kommas getrennte Listen).
  • Tupel sehen immer wie Tupel aus und alles, was wie ein Tupel aussieht, ist ein Tupel.

Pattern-Matching

<th>
  Reason
</th>

<td>
  <pre><code>let res = switch (x) {

| A((x, y)) => switch (y) { | None => 0 | Some(i) => 10 } | B((x, y)) => 0 };

OCaml 
let res = match x with
  | A (x, y) -> match y with
    | None -> 0
    | Some i -> 10
  | B (x, y) -> 0

Kannst du den Fehler im OCaml-Beispiel erkennen? Dies ist einer der häufigsten Fehler bei OCaml-Programmierern. Das verschachtelte match muss von Klammern umgeben werden, ansonsten wird der Some-Fall so geparst, als würde er zum äußeren match gehören. Veranschaulicht ist es eigentlich:

let res = match x with
  | A (x, y) -> match y with
    | None -> 0
    | Some i -> 10
    | B (x, y) -> 0

Reason verhindert dieses Problem durch die erzwungenen {} um die switch-Fälle.

Verschiedene Verbesserungen

OCaml verlangt keine Klammern um Sequenzen (a;b;c;d) oder Tupel (x,y), was somit dazu führt, dass eine Menge anderer sehr praktischer Syntaxregeln ausgeschlossen werden. Da Reason immer {} verwendet, um Sequenzen oder let-Bindungen zu umgeben, und Reason immer () um Tupel herum verlangt, werden viele andere Syntaxkonstrukte intuitiver ausgedrückt, ohne zusätzliche Umfassung mit Klammern.

Lambdas als Record-Felder brauchen nicht mehr extra Klammern

Dies ist eine willkommene Verbesserung weil die Fehler bei OCaml-Typen, die ein Benutzer sehen würde, sehr verwirrend sind, wenn der Rückgabewert der Funktion ein Tupel mit dem Infix-Komma , wäre.

Verschiedene Unterschiede

Vorrang von as

Mit Reason hat as eine höhere Priorität als |. Dies ermöglicht das Erstellen von as-Aliasen für ganze Zeilen im Pattern-Matching.

<th>
  Reason
</th>

<td>
  <pre><code>let ppp =

switch (MyThing(20)) { | MyThing(x) as ppp | YourThing(x) as ppp => ppp };

OCaml 
let ppp = match MyThing 20 with
  | (MyThing x as ppp)
  | (YourThing x as ppp) -> ppp;

Änderungen mutierbarer Record-Felder

Da Gleichheiten und ihre Negationen in Reason konsistenter gemacht wurden, ist der Operator = für die Aktualisierung mutierbarer Felder verfügbar.

OCamlReason
myRec.field <- "next"myRec.field = "next"

Präfix-Operatoren

Reasons ! wird für boolean not verwendet. Dereferenzierung verwendet das postfix ^.

OCamlReason
let x = !(foo.bar)let x = foo.bar^;
let x = !(foo#bar)let x = foo#bar^;
let x = !(!foo.bar)let x = foo^.bar^;
let x = !(!foo#bar)let x = (foo^)#bar^;
let x = !(!(foo.bar))let x = foo.bar^ ^;
let x = !(!(foo#bar))let x = foo#bar^ ^;
let x = !!(foo.bar)let x = !!foo.bar;
let x = !!(foo#bar)let x = !!foo#bar;
let x = !~(foo.bar)let x = !~foo.bar;
let x = !~(foo#bar)let x = !~foo#bar;

Maskierung in Kommentaren

Da Reason Kommentare im C-Stil verwendet, müssen einige obskure benutzerdefinierte Präfix- / Infix-Operatoren anders geschrieben werden. Die Regeln für Präfix / Infix-Operatoren sind die gleichen wie in der OCaml-Syntax, jedoch mit den folgenden Ausnahmen:

Wenn ein beliebiges Zeichen außer dem ersten in einem Präfix / Infix-Operator ein Stern oder ein Schrägstrich ist, muss das zuerst mit einem Backslash maskiert werden. Diese werden ohne den Schrägstrich geparst, wenn es dem AST hinzugefügt wird. Bei der erneuten Ausgabe werden die Schrägstriche zur Maskierung automatisch hinzugefügt.

OCamlReason
let (/*) a b = a + blet (/\*) = (a, b) => a + b;
let x = 12 /-* 23 /-* 12let x = 12 /-* 23 /-* 12;
let y = (/*) a blet y = a /\* b;
let (!=*) q r = q + rlet ( !=* ) = (q, r) => q + r;
let res = q (!=*) rlet res = q(( !=* ), r);
let (!=/*) q r = q + rlet ( !=/\* ) = (q, r) => q + r;
let res = q (!=/*) rlet res = q(( !=/\* ), r);

Umbenennung von Operatoren

Wenn Reason == benutzt, um OCamls = zu repräsentieren, und === einsetzt, um OCamls == darzustellen, wie würde dann Reason das ===-Symbol von OCaml repräsentieren (wenn es definiert wäre)? Reason stellt einen Weg bereit! „Maskiere“ das dreifache Gleichheitszeichen!

IdentifierBedeutungOCamlReason
"==="Benutzerdefinierter Wertx === yx \=== y
