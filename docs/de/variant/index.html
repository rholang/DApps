Variant!

Erblicke das Kronjuwel der Reason-Datenstrukturen!

Bei den meisten Datenstrukturen in den meisten Sprachen geht es um „dies und das“. Eine Variante erlaubt uns, „dies oder das“ auszudrücken.

type myResponseVariant =
  | Yes
  | No
  | PrettyMuch;

let areYouCrushingIt = Yes;

Yes, No und PrettyMuch sind weder Strings noch Referenzen, noch irgendein spezieller Datentyp. Sie werden „Konstruktoren“ (oder „Tags“) genannt. Der senkrechte | Strich trennt die Konstruktoren voneinander.

Bemerkung: die Konstruktoren einer Variante müssen mit einem Großbuchstaben beginnen.

Verwendung

Zusammen mit einer Variante kommt eine der wichtigsten Funktionen von Reason, der switch-Ausdruck.

In Reason ähnelt ein switch visuell dem switch in anderen Sprachen (auch bekannt als ein großes if/elseif/elseif...). Es erlaubt dir, jeden möglichen Fall einer Variante zu prüfen. Um es zu verwenden, liste jeden Variantenkonstruktor der bestimmten Variante auf, die du verwenden möchtest, gefolgt von einem => und dem entsprechenden Ausdruck.

let message =
  switch (areYouCrushingIt) {
  | No => "No worries. Keep going!"
  | Yes => "Great!"
  | PrettyMuch => "Nice!"
  };
/* message is "Great!" */

Eine Variante hat eine sehr reiche Menge an Hilfestellung durch das Typsystem. Zum Beispiel geben wir dir einen Typfehler, wenn du vergessen hast, einen Fall deiner Variante abzudecken, oder wenn zwei Fälle redundant sind. Achte darauf, dass du dir switch und Pattern-Matching in einem späteren Abschnitt ansiehst!

Varianten brauchen eine explizite Definition

Wenn sich die von dir verwendete Variante in einer anderen Datei befindet, bringe sie in den Sichtbarkeitsbereich, den du für ein Record verwenden würdest:

/* Zoo.re */

type animal = Dog | Cat | Bird;

/* example.re */

let pet: Zoo.animal = Dog; /* preferred */
/* or */
let pet = Zoo.Dog;

Konstruktorargumente

Die Konstruktoren einer Variante können weitere Daten durch Kommatrennung enthalten.

type account =
  | None
  | Instagram(string)
  | Facebook(string, int);

Hier enthält Instagram einen string, und Facebook enthält einen string und ein int. Verwendung:

let myAccount = Facebook("Josh", 26);
let friendAccount = Instagram("Jenny");

Fällt dir auf, wie die Verwendung eines Konstruktors einem Funktionsaufruf ähnelt? Es ist als wäre Facebook eine Funktion, die zwei Argumente erhält. Dies ist kein Zufall; Es gibt einen Grund, warum die Daten eines Konstruktors „Konstruktorargumente“ genannt werden.

Mit switch kannst du auf die Argumente eines Konstruktors Pattern-Matching anwenden (nochmals, wie in einem späteren Abschnitt erklärt):

let greeting =
  switch (myAccount) {
  | None => "Hi!"
  | Facebook(name, age) => "Hi " ++ name ++ ", you're " ++ string_of_int(age) ++ "-year-old."
  | Instagram(name) => "Hello " ++ name ++ "!"
  };

Lobende Erwähnungen

Die Standardbibliothek stellt zwei wichtige Varianten bereit, die dir noch oft begegnen werden.

option

type option('a) = None | Some('a);

Dies ist die Konvention um einen „Null-fähigen“ (auch bekannt als undefined oder null) Wert in anderen Sprachen zu simulieren. Dank dieser bequemen Typdefinition kann Reason standardmäßig für jeden Wert Null vermeiden. Ein int wird immer ein int sein, niemals „int oder null oder undefined“. Falls du einen „int mit möglichem Null-Wert“ ausdrücken möchtest, dann würdest du option(int) verwenden, dessen mögliche Wert None und Some(int) sind. switch zwingt dich, beide Fälle zu behandeln. Daher hat ein reines Reason-Programm keine Null-Fehler.

list

type list('a) = Empty | Head('a, list('a));

Nicht die eigentliche Typdefinition. Nur eine Andeutung.

Dies drück aus: „eine Liste, die einen Wert des Typs a enthält (was auch immer das ist), ist entweder leer oder enthält den Wert und eine weitere Liste“.

Reason gab list syntaktischen Zucker. [1, 2, 3] ist konzeptuell äquivalent mit Head(1, Head(2, Head(3, Empty))). Nochmals, switch zwingt dich, jeden Fall dieser Variante zu behandeln, inklusive Empty (auch als [] bekannt). Dies eliminiert eine weitere große Kategorie von Zugriffsfehlern.

Andere Typen, die Varianten ähneln

Wusstest du, dass du switch für string, int, float, array und die meisten anderen Datenstrukturen verwenden kannst? Probier es aus!

Tips & Tricks

Pass auf, dass du einen Konstruktor mit zwei Argumenten nicht mit einem Konstruktor verwechselst, der ein Tupel als einziges Argument hat:

type account =
  | Facebook(string, int) /* 2 arguments */;
type account2 =
  | Instagram((string, int)) /* 1 argument - happens to be a 2-tuple */;

Varianten müssen Konstruktoren haben

Wenn du von einer untypisierten Sprache kommst, dann könntest du versucht sein, type foo = int | string auszuprobieren. Dies ist nicht möglich in Reason; Du müsstest jedem Zweig einen Konstruktor geben: type foo = Int(int) | String(string). Obwohl dies normalerweise ein Anti-Pattern sein könnte. Der Abschnitt zu Designentscheidungen unten erläutert dies ausführlicher.

Interoperabilität mit JavaScript

Dieser Abschnitt setzt Kenntnis über BuckleScripts FFI voraus. Überspringe diesen, wenn du noch nicht das Bedürfnis hast, Varianten zum Einbinden von JS-Funktionen zu verwenden.

Einige JS-Bibliotheken verwenden Funktionen, die viele Arten von Argumenten akzeptieren können. In diesen Fällen ist es sehr verlockend, sie als Varianten zu modellieren. Nehmen wir beispielsweise an, es gäbe eine JS-Funktion myLibrary.draw, die entweder ein number- oder ein string-Argument akzeptiert. Du könntest versucht sein, es so zu binden:

/* reserved for internal usage */
[@bs.module "myLibrary"] external draw : 'a => unit = "draw";

type animal =
  | MyFloat(float)
  | MyString(string);

let betterDraw = (animal) =>
  switch (animal) {
  | MyFloat(f) => draw(f)
  | MyString(s) => draw(s)
  };

Du könntest dies definitiv tun, aber es gibt bessere Wege! Zum Beispiel einfach zwei externals, die beide zum gleichen JS-Aufruf kompiliert werden:

[@bs.module "myLibrary"] external drawFloat : float => unit = "draw";
[@bs.module "myLibrary"] external drawString : string => unit = "draw";

BuckleScript stellt auch ein paar andere Wege bereit, um dies zu erreichen.

Variantentypen werden durch Feldnamen gefunden

Siehe dazu bitte diesen Abschnitt zu Record. Varianten sind genau so: eine Funktion kann keinen willkürlichen Konstruktor annehmen, der in zwei verschiedenen Varianten vorkommt. Noch einmal, so ein Feature existiert und heißt polymorphe Varianten. Wir werden darüber in der Zukunft sprechen =).

Designentscheidungen

Varianten, in ihren vielfältigen Formen (polymorphe Varianten, offene Varianten, GADT, usw.), sind womöglich das Feature eines Typsystem wie Reasons. Die oben erwähnte option-Variante zum Beispiel macht die Notwendigkeit von Null-fähigen Typen unnötig, die eine Hauptursache für Fehler in anderen Sprachen darstellen. Philosophisch gesehen besteht ein Problem aus vielen möglichen Verzweigungen / Bedingungen. Falsche Behandlung dieser Bedingungen macht die Mehrheit von dem aus, was wir Bugs nennen. Ein Typsystem beseitigt keine Fehler auf magische Weise. Es weist auf die unbehandelten Bedingungen hin und bittet dich, sie zu behandeln*. Die Fähigkeit, „dies oder das“ korrekt zu modellieren, ist entscheidend.

Zum Beispiel wundern sich einige Leute, wie das Typsystem schlecht formatierte JSON-Daten sicher davon abhalten kann, sich in ihrem Programm zu verbreiten. Sie können es nicht, zumindest von selbst! Wenn der Parser den option-Typ None | Some(actualData) zurückgibt, dann müsstest du den Fall None in späteren Aufrufszenarien explizit behandeln. Das ist alles, was es gibt.

In Bezug auf Performance kann eine Variante die Programmlogik enorm beschleunigen. Hier ist ein Stück JavaScript:

let data = 'dog';
if (data === 'dog') {
  ...
} else if (data === 'cat') {
  ...
} else if (data === 'bird') {
  ...
}

Hier wird eine lineare Anzahl von Verzweigungen überprüft (O(n)). Vergleiche dies mit einer Reason-Variante:

type animal = Dog | Cat | Bird;
let data = Dog;
switch (data) {
| Dog => ...
| Cat => ...
| Bird => ...
}

Der Compiler sieht die Variante und

  1. ändert sie konzeptuell in type animal = 0 | 1 | 2 ab
  2. übersetzt switch in ein Format von konstanter Zeit (O(1)).

Du fragst dich vielleicht, warum typisierte funktionale Sprachen so oft zum Parsen verwendet werden; in einem großen Baum effizient und sicher zu wechseln ist für Varianten das bestmögliche Szenario.

Schon verblüfft? Varianten haben eine tiefe Verbindung zu anderen Gebieten der Mathematik; Siehe hier für eine interessante Erkundung.

* Es ist immer schöner, das Problem durch einen besseren Entwurf zu lösen, als auf ein Typsystem zurückzugreifen, um die Fallstricke abzudecken. In der Realität ist es unrealistisch, dies für jedes Problem zu tun oder einfach nur jedes Problem vollständig zu verstehen, um eine Lösung zu entwickeln. Ein Typsystem ermöglicht es dir, eine große Kategorie von Änderungen an Codebasen vorzunehmen, ohne das Ganze im Voraus verstehen zu müssen. Dies ist großartig für geleitete Erkundungen. In dieser Hinsicht erlauben Typen uns, eine API nicht übermäßig zu entwerfen, nur um die einfachen Fallstricke der Aufrufszenarien zu umgehen. Sie reduzieren die Schichten von Abstraktionen, die nötig sind, um „Dinge zu erledigen“, was wiederum die kognitive Belastung in Aufrufszenarien reduziert.
