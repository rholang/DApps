Interop

Tan solo escribe JavaScript en medio de tu código Reason

If you're just hacking things together, this can be very nice, but you also have all of the unsafety of JavaScript code

Js.log("this is reason");

[%bs.raw {| console.log('here is some javascript for you') |}];

{| and |} son los delimitadores de una cadena OCaml en multiples líneas. También puedes poner una etiqueta ahí e.g. {something| y entonces buscará una coincidencia |something} de cierre.

Y este es el JavaScript resultante:

// Generado por BUCKLESCRIPT VERSION 1.7.4, POR FAVOR EDITAR CON CUIDADO
'use strict';
console.log("esto es reason");
console.log('aquí hay algo de javascript para ti');

Poniendo algo de JavaScript, y haciéndolo accesible desde Reason

¿Qué pasa si quieres un valor que pueda ser usado desde tu código Reason?

Js.log("esto es reason");
let x = [%bs.raw {| 'aquí un string de javascript' |}];
Js.log(x ++ " de vuelta en tierra de reason"); /* ++ es el operador para concatenar tipos string */

Ahora quizás te preguntes "¿qué magia es esta?" ¿Cómo sabia ocaml que x era una cadena?" No es así. ¡El tipo de x en este código es un tipo mágico que se unificará con todo! Esto es bastante peligroso y puede tener efectos en cascada en el algoritmo de inferencia de tipos de OCaml.

let y = [%bs.raw {| 'algo' |}];
Js.log(("un string" ++ y, 10 + y));
/* ¡¡cuidado!! ocaml no impedirá que uses y como 2 tipos totalmente diferentes */

Para solucionar esto, siempre debes proporcionar un tipo concreto para el resultado de bs.raw.

let x: string = [%bs.raw {| 'bien tipado' |}];
Js.log(x ++ " de vuelta en tierra de reason");
/* ocaml generará un error si intentas usar x como algo distinto a un string */

¡Y aquí está la salida!

// Generado por BUCKLESCRIPT VERSION 1.7.4, POR FAVOR EDITAR CON CUIDADO
'use strict';
console.log("esto es reason");
var x = ( 'aquí un string de javascript' );
console.log(x + " de vuelta en tierra de reason");
var y = ( 'algo' );
console.log(/* tupla */[
      "un string" + y,
      10 + y | 0
    ]);
var x$1 = ( 'bien tipado' );
console.log(x$1 + " de vuelta en tierra de reason");

¡La diferencia entre los 2 %% de la anterior sección y el 1 % aquí es importante! [%%something...] es un "punto de extensión" ("extension point") de OCaml que representa una declaración de nivel superior (no se puede mostrar dentro de una función o valor, por ejemplo). [%something...] es un punto de extensión que substituye una expresión y se puede poner casi en cualquier lugar -- ¡pero asegúrate de que el JavaScript que pones dentro es realmente una expresión! Por ejemplo, no pongas un punto y coma después de él, u obtendrás un error de sintaxis cuando intentes ejecutar el JavaScript resultante.

Insertando en una función y pasanso valores

Necesitaremos saber un poco de la representación de varios valores que Bucklescript hace en tiempo de ejecución para este trabajo.

  • strings son cadenas de caracteres, ints y floats son sólo números
  • una Array es una lista de longitud fija mutable en OCaml y se representa como un array simple JavaScript.
  • un List es una lista encadenada inmutable de estilo funcional y es definitivamente lo más idiomático para utilizar en la mayoría de los casos. However, its representation is more complicated (try Js.log([1,2,3,4]) to check it out). Debido a esto, generalmente convierto desde y hacia Arrays cuando estoy hablando con JavaScript, a través de Array.of_list y Array.to_list.
  • Si desea profundizar más, existe una lista en la documentación de BuckleScript

Sabiendo esto, podemos escribir una función en JavaScript que sólo acepte un array y devuelve un número, sin mucha dificultad.

let jsCalculate: (array(int), int) => int = [%bs.raw
  {|
 function (numbers, scaleFactor) {
   var result = 0;
   numbers.forEach(number => {
     result += number;
   });
   return result * scaleFactor;
 }
|}
];

let calculate = (numbers, scaleFactor) => jsCalculate(Array.of_list(numbers), scaleFactor);

Js.log(calculate([1, 2, 3], 10)); /* -> 60 */

Por supuesto, esta función que he escrito en JavaScript podría ser portada a Reason sin muchas molestias.

Recuerda que se trata de una solución puntual que es muy útil para aprender y así poder empezar rápidamente y hacer algo, pero un buen ejercicio es volver atrás y convertirlo de nuevo en código tipado y seguro de Reason.

I've run into more than a few bugs because of raw JavaScript that I added to save time

Asentarse y ser disciplinado con las cosas

Hasta ahora hemos estado usando bs.raw, que es una manera muy rápida y laxa de hacerlo y no conveniente para producción.

Pero ¿qué pasa si realmente necesitamos llamar a una función en JavaScript? Esto es necesario para interactuar con el DOM, o usar módulos de node. En BuckleScript, se utiliza una declaración external (documentación).

Obtener un valor y obtener una función es en ambos casos muy fácil:

[@bs.val] external pi : float = "Math.PI";
let tau = pi *. 2.0;
[@bs.val] external alert : string => unit = "alert";
alert("hola");

Pero ¿qué pasa cuando queremos algo más complicado? Así es como podríamos llamar a getContext en un nodo DOM de Canvas:

type canvas;

type context;

/* we're leaving these types abstract, because we won't
 * be using them directly anywhere */
[@bs.send] external getContext : (canvas, string) => context = "getContext";

let myCanvas: canvas = [%bs.raw {| document.getElementById("mycanvas") |}];

let ctx = getContext(myCanvas, "2d");

Expliquemos lo que está sucediendo. Hemos creado algunos tipos abstractos para el nodo DOM de Canvas y el objeto RenderingContext asociado.

Luego hemos hecho una función getContext, pero en vez de @bs.val usamos @bs.send y utilizamos una cadena vacía para el texto del external. @bs.send significa "Estamos llamando un método de el primer argumento", que en este caso es Canvas. Dado lo anterior, BuckleScript se traducirá getContext (theFirstArgument, theSecondArgument) a theFirstArgument.getContext(theSecondArgument, ...).

La cadena vacía significa "el nombre JS es el mismo que el nombre que estamos dando al external en BuckleScript" - en este caso getContext. Si quisiéramos llamarle de otra manera (como getRenderingContext), tendríamos que suministrar la cadena "getContext" así BuckleScript llamará a la función correcta.

Añadamos una función más solo porque es interesante.

[@bs.send] external fillRect : (context, float, float, float, float) => unit = "fillRect";

¡Y ahora podemos dibujar algo!

fillRect(ctx, 0.0, 0.0, 100.0, 100.0);

No es mucho, pero agregar otros métodos del canvas es similar, y entonces puedes comenzar haciendo algunas cosas realmente divertidas.

Así que ¿como es el JavaScript compilado?

'use strict';
var tau = Math.PI * 2.0;
alert("hello");
var myCanvas = ( document.getElementById("mycanvas") );
var ctx = myCanvas.getContext("2d");
ctx.fillRect(0.0, 0.0, 100.0, 100.0);

¡Wow! ¿Ves cómo BuckleScript solo introdujo nuestra variable pi por nuestra cuenta? Y el resultado es casi exactamente como si fuera escrito a mano.

Usando librerías JavaScript existentes

When folks write wrappers for a particular JavaScript library, they'd usually publish it to npm. Head over to the Libraries to find out how to find these.

To use a library that does not have existing wrappers, however, you'll want to first install the npm package as usual, e.g. using npm install --save <package-name>, then just go ahead and write your wrapper. Probablemente encontrarás la herramienta FFI de bs.module particularmente útil; esta emite los imports o requires correctos, dependiendo del formato de compilación del módulo de destino de JS.

As an example, here's the entire source code of the bs.glob wrapper (converted to Reason, the original is OCaml):

type error;

[@bs.module] external glob : (string, (Js.nullable(error), array(string)) => unit) => unit = "glob";

[@bs.val] [@bs.module "glob"] external sync : string => array(string) = "glob";

Y las partes relevantes de package.json:

{
  "name": "bs-glob",
  "version": "0.1.0",
  ...
  "devDependencies": {
    "bs-platform": "^1.9.1"
  },
  "dependencies": {
    "glob": "^7.1.2"
  }
}
