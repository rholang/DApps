Variant!

¡He aquí, la joya de la corona de las estructuras de datos de Reason!

La mayoría de las estructuras de datos en la mayoría de lenguajes son sobre "esto y eso". Una variante nos permite expresar "esto o eso".

type myResponseVariant =
  | Yes
  | No
  | PrettyMuch;

let areYouCrushingIt = Yes;

Yes, No and PrettyMuch no son strings, o referencias, y tampoco algún tipo de datos especial. Ellos son llamados "constructores" (o "etiquetas"). La barra | separa cada constructor.

Nota: un constructor de variante necesita estar en mayúsculas.

Uso

Junto con una variante viene una de las características más importantes de Reason, la expresión switch.

Un switch en Reason es visualmente similar a el switch de otros lenguajes (conocido como un gran if/elseif/elseif...). Esto te permite comprobar cada posible caso de una variante. Para usarlo, enumera cada constructor de variante de la variante en particular que te gustaría usar, cada uno seguido de un => y la expresión correspondiente a ese caso.

let message =
  switch (areYouCrushingIt) {
  | No => "No te preocupes. ¡Sigue intentando!"
  | Yes => "¡Genial!"
  | PrettyMuch => "¡Muy bien!"
  };
/* mensaje es "¡Genial!" */

Una variante tiene una cantidad extremadamente rica de asistencia del sistema de tipos. Por ejemplo, te daremos un error de tipo si has olvidado cubrir un caso de tu variante, o si dos casos son redundantes. ¡Asegúrate de revisar switch y el patrón de coincidencia pattern-matching en una sección posterior!

Las Variantes Necesitan una Definición Explícita

Si la variante que estas usando está en un archivo diferente, incorporala dentro del ámbito como lo harías con un registro:

/* Zoo.re */

type animal = Dog | Cat | Bird;

/* example.re */

let pet: Zoo.animal = Dog; /* preferred */
/* o */
let pet = Zoo.Dog;

Argumentos de constructor

Los constructores de variantes pueden tener datos extra separados por comas.

type account =
  | None
  | Instagram(string)
  | Facebook(string, int);

Aquí, Instagram contiene un string, y Facebook contiene un string y un int. Uso:

let myAccount = Facebook("Josh", 26);
let friendAccount = Instagram("Jenny");

¿Observastes como usando un constructor es como llamar a una función? Eso como si Facebook fuese una función que acepta dos argumentos. Esto no es una coincidencia; hay una razón de por qué los datos del constructor se llaman "argumentos de constructor".

Usando switch, puedes aplicar la coincidencia de patrones pattern-match (de nuevo: descrito en una sección posterior) a argumentos de constructor:

let greeting =
  switch (myAccount) {
  | None => "¡Hola!"
  | Facebook(name, age) => "Hola " ++ name ++ ", tú tienes " ++ string_of_int(age) ++ " años de edad."
  | Instagram(name) => "¡Hola " ++ name ++ "!"
  };

Menciones Honorables

La biblioteca estándar expone dos variantes importantes que escucharás a menudo.

opción

type option('a) = None | Some('a);

Esto es la convención usada para simular un valor "nullable" (es decir, undefined o null) en otros lenguajes. Gracias a esta práctica definición de tipo, Reason puede establecer que por defecto cada valor sea no-nullable. Un int siempre será un int, nunca "int o null o undefined". Si quieres expresar un "int nullable", usarías option(int), cuyos valores posibles son None o Some(int). switch te obliga a manejar ambos casos; por lo tanto, un programa puro Reason no tiene errores null.

lista

type list('a) = Empty | Head('a, list('a));

No la definición de tipo real. Sólo una ilustración.

Esto dice: "una lista que contiene un valor de tipo a (sea lo que sea) esta vacío o tiene ese valor además de otra lista".

Reason le dio a list un endulzamiento sintáctico syntax sugar. [1, 2, 3] es conceptualmente equivalente a Head(1, Head(2, Head(3, Empty))). Una vez más,, switch te obliga a manejar cada caso de esta variante, incluyendo Empty (también conocido como []). Esto elimina otra gran categoría de fallos de acceso.

Otros tipos similares a variantes

¿Sabías que puedes usar switch en string, int, float, array y la mayoría de otras estructuras de datos? ¡Pruébalo!

Consejos y trucos

Ten cuidado y no confundas un constructor que lleva 2 argumentos con un constructor que lleva un sólo argumento tupla:

type account =
  | Facebook(string, int) /* 2 argumentos */;
type account2 =
  | Instagram((string, int)) /* 1 argumento - resulta ser una 2-tupla */;

Las variantes Deben Tener constructores

Si vienes de un lenguaje sin tipos, podrías estar tentado de intentar type foo = int | string. Esto no es posible en Reason; tendrías que dar a cada rama un constructor: type foo = Int(int) | String(string). Aunque normalmente, necesitar esto podría ser un anti-patrón. La sección de abajo sobre Decisiones de Diseño explica más.

Interoperabilidad con JavaScript

Esta sección asume que se posee conocimiento sobre FFI de BuckleScript. Skip this if you haven't felt the itch to use variants for wrapping JS functions yet.

Bastantes librerías JS usan funciones que pueden aceptar muchos tipos de argumentos. En esos casos, es muy tentador modelarlos como variantes. Por ejemplo, supón que hay una función JS myLibrary.draw que toma ya sea un number o un string. Podrías estar tentado de enlazarlos bind así:

/* reservado para uso interno*/
[@bs.module "myLibrary"] external draw : 'a => unit = "draw";

type animal =
  | MiFloat(float)
  | MiString(string);

let betterDraw = (animal) =>
  switch (animal) {
  | MiFloat(f) => draw(f)
  | MiString(s) => draw(s)
  };

Podrías definitivamente hacer eso, pero ¡hay mejores maneras! Por ejemplo, simplemente dos external que compilen a la misma llamada JS:

[@bs.module "myLibrary"] external drawFloat : float => unit = "draw";
[@bs.module "myLibrary"] external drawString : string => unit = "draw";

BuckleScript also provides a few other ways to do this.

Los Tipos de Variantes Son Encontrados Por Nombre de Campo

Por favor, diríjase a esta sección sobre registros (records). Las variantes son igual: una función no puede aceptar un constructor arbitrario compartido por dos variantes diferentes. Otra vez, tal característica existe, se llama variante polimorfa. Hablaremos de esto en el futuro =).

Decisiones de Diseño

Variantes, en sus muchas formas (variante polimorfa, variante abierta, GADT, etc), son probablemente la característica de un sistema de tipos como el de Reason. La variante option mencionada anteriormente, por ejemplo, elimina la necesidad de usar tipos null (nullable), una gran fuente de fallos en otros lenguajes. Filosóficamente hablando, un problema es compuesto de muchas posibles ramas/condiciones. Manejar de forma inadecuada esas condiciones es la mayoría de lo que llamamos bugs. Un sistema de tipos no elimina mágicamente los fallos; te señala las condiciones sin gestionar y te pide que las cubras*. La habilidad de modelar "esto o aquello" correctamente es crucial.

Por ejemplo, algunas personas se preguntan como el sistema de tipos puede eliminar de forma segura datos JSON con muy mal formato que se propaguen en sus programas. ¡No lo hacen, no por ellos mismos! Pero si el analizador sintáctico devuelve el tipo option, None | Some(actualData), entonces tendrías que gestionar el caso None explícitamente en llamadas posteriores. Eso es todo lo que hay.

En cuestión de rendimiento, una variante puede potencialmente aumentar tremendamente tu lógica de programa. Aquí hay un trozo de JavaScript:

let data = 'perro';
if (data === 'perro') {
  ...
} else if (data === 'gato') {
  ...
} else if (data === 'gato') {
  ...
}

Hay una cantidad lineal de comprobación de ramas aquí (O(n)). Compara esto al usado en una variante Reason:

type animal = Dog | Cat | Bird;
let data = Dog;
switch (data) {
| Dog => ...
| Cat => ...
| Bird => ...
}

El compilador ve la variante, y entonces

  1. conceptualmente los convierte en type animal = 0 | 1 | 2
  2. compila switch a un formato de tiempo constante (O(1)).

Te podrías preguntar por qué los lenguajes funcionales tipados son usados tan a menudo para análisis: utilizar switch en un árbol grande de forma eficiente y segura es el mejor escenario para variantes.

¿Sorprendido ya? Las variantes tienen una profunda conexión con otros campos de las matemáticas; Mira aquí para una exploración interesante.

* Siempre es más agradable diseñar para resolver el problema que recurrir a un sistema de tipos que cubra las deficiencias; En la práctica, es poco realista hacerlo para cada problema, o incluso solo para entender completamente cada problema a fin de diseñar una solución. Un sistema de tipos te permite hacer con seguridad una gran categoría de cambios a bases de código sin necesidad de entender todo por adelantado. Esto es genial para exploración guiada. En este sentido, los tipos también nos permiten no necesitar diseñar en exceso una API solo para eludir las deficiencias de los interlocutores callers. Ellos reducen las capas de abstracciones necesarias para "finalizar las cosas", lo que en retorno reduce la carga cognitiva de los interlocutores callers.
