Record

Los registros son como objetos JavaScript pero son

  • más ligeros
  • inmutables por defecto
  • fijados en nombres de campo y tipos
  • muy rápidos
  • de tipo un poco más rígido

Uso

Tipo (obligatorio):

type person = {
  age: int,
  name: string
};

Valor (esto se infiere ser de tipo person):

let me = {
  age: 5,
  name: "Big Reason"
};

Acceso (la notación familiar punto):

let name = me.name;

Los registros necesitan una Definición Explícita

Si sólo escribes {age: 5, name: "Bebé Reason"} sin una declaración explícita en algún lugar por encima, el sistema le dará un error. Si la definición del tipo se encuentra en otro archivo, debe indicar explícitamente qué archivo es:

/* School.re */

type person = {age: int, name: string};

/* example.re */

let me: School.person = {age: 20, name: "Gran Reason"};
/* or */
let me = School.{age: 20, name: "Gran Reason"};
/* or */
let me = {School.age: 20, name: "Gran Reason"};

Cualquiera de los 3 de arriba dice "la definición de este registro se encuentra en el archivo School". La primero, la anotación de tipo regular, es preferida.

Actualización inmutable

Nuevos registros pueden crearse de registros antiguos con el operador de expansión .... El registro original no es mutado.

let meNextYear = {...me, age: me.age + 1};

¡Esta actualización es muy eficiente! Haz unas pruebas en nuestro playground para ver cómo se compilan los registros.

Nota: la expansión no puede añadir nuevos campos, por que la forma de un registro es fija en base a su tipo.

Actualización mutable

Los campos de registros pueden ser mutables opcionalmente. Esto te permite actualizar los campos directamente con el operador =.

type person = {
  name: string,
  mutable age: int
};
let baby = {name: "Baby Reason", age: 5};
baby.age = baby.age + 1; /* alter `baby`. Happy birthday! */

Sintaxis abreviada

Para reducir la redundancia, proporcionamos punning para los tipos y valores de un registro. Punning se refiere a la forma abreviada de sintaxis que se puede usar cuando el nombre de un campo coincide con el nombre de su valor/tipo:

type horsePower = {power: int, metric: bool};

let metric = true;
let someHorsePower = {power: 10, metric};
/* same as the value {power: 10, metric: metric}; */

type car = {name: string, horsePower};
/* same as the type {name: string, horsePower: horsePower}; */

¡Tenga en cuenta que no existe punning para un solo campo de registro! {foo} no hace lo que piensas (es un bloque que devuelve el valor de foo).

Consejos y trucos

Interoperabilidad con JavaScript

Si trabajas con JavaScript, la sintaxis de registro y operaciones deben ser familiares, y puedes estar tentado de interoperar con JS mediante la conversión de un objeto JS a un registro y viceversa. Está bien, pero tenemos una manera mejor sin sobrecarga de conversión! See here

Los tipos de registro son encontrados por el nombre del campo

With records, you cannot say "I'd like this function to take any record type, as long as they have the field age". The following works, but not as expected:

type person = {age: int, name: string};
type monster = {age: int, hasTentacles: bool};

let getAge = (entity) => entity.age;

The last line's function will infer that the parameter entity must be of type monster. The following code's last line fails:

let kraken = {age: 9999, hasTentacles: true};
let me = {age: 5, name: "Baby Reason"};

getAge(kraken);
getAge(me);

The type system will complain that me is a person, and that getAge only works on monster. If you need such capability, use Reason objects, described here.

Decisiones de diseño

After reading the constraints in the previous sections, and if you're coming from a dynamic language background, you might be wondering why one would bother with record in the first place instead of straight using object, since the former needs explicit typing and doesn't allow different records with the same field name to be passed to the same function, etc.

  1. La verdad es que la mayoría de las veces en tu aplicación, la forma de tus datos es en realidad fija, y si no, puede ser potencialmente representada mejor como una combinación de variante (introducida próximamente) + registro*.
  2. Los registros, como sus campos son fijos, son compilados a un array con indice de acceso a arrays en vez de un objeto JS (¡ pruebalo en el playground!). En nativo, compila a basicamente una region de memoria donde un campo de acceso es solo un campo de búsqueda + un acceso real, conocido como 2 instrucciones de ensamblador. Los buenos días pasados cuando se media en nanosegundos...
  3. Finalmente, como un tipo de registro es resuelto buscando esa única declaración de tipo explícita (nosotros llamamos a esto "tipado nominal"), los mensajes de error de tipo son mejores que sus equivalentes ("tipado estructural", como para tuplas). Esto hace la refactorización mas fácil; cambiar los campos de tipos naturalmente permite al compilador saber que es todavía el mismo registro, solo mal utilizado en algunas partes. De otra manera, bajo tipado estructural, seria difícil decir cuando el lugar de definición o el lugar de uso es incorrecto.

* And we're not just finding excuses for ourselves! Reason objects do support these features.
