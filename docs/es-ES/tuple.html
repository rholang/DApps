Tuple

Las Tuplas son

  • inmutables
  • ordenadas
  • de tamaño fijo en el momento de creación
  • heterogéneas (puede contener diferentes tipos de valores)
let ageAndName = (24, "Lil' Reason");
let my3dCoordinates = (20.0, 30.5, 100.0);

Los tipos de tuplas pueden utilizarse también en las anotaciones de tipo. Los tipos tupla se asemejan visualmente a valores de tuplas.

let ageAndName: (int, string) = (24, "Lil' Reason");
/* a tuple type alias */
type coord3d = (float, float, float);
let my3dCoordinates: coord3d = (20.0, 30.5, 100.0);

Nota: no existe tupla de tamaño 1. Sólo usas el valor mismo.

Uso

La biblioteca estándar proporciona fst y snd (aquí, debajo "Operaciones emparejadas"), funciones convenientes que te permiten acceder al primer y segundo elemento de una 2-tupla. Generalmente, accederías a los miembros de una n-tupla a través de la desestructuración (descrito más adelante en la barra lateral):

let (_, y, _) = my3dCoordinates; /* now you've retrieved y */

El carácter _ significa que estas ignorando los miembros indicados de la tupla.

Las tuplas no están pensadas para ser actualizadas mutatívamente; crearás nuevas por desestructuración de las antiguas.

Consejos y trucos

Usarás las tuplas en situaciones prácticas en las que pasar multiples valores sin demasiada ceremonia. Por ejemplo, para devolver muchos valores:

let getCenterCoordinates = () => {
  let x = doSomeOperationsHere();
  let y = doSomeMoreOperationsHere();
  (x, y)
};

O para el "patrón de coincidencia" (cubierto más adelante) en el conjunto de posibilidades:

switch (isWindowOpen, isDoorOpen) { /* this is a 2-tuple */
| (true, true) => ...
| (true, false) => ...
| (false, true) => ...
| (false, false) => ...
}

Tratar de mantener el uso de tuplas local. Para estructuras de datos que son de larga vida y se pasan a menudo, elige un registro, que cuenta con nombre de campos.

Un tipo de tupla puede también ser llamado un "tipo de producto", y (string, int) se escribe como string * int en algunas partes. La idea es que una tupla es un "producto cartesiano"; imaginar una cuadrícula 2D, ¡con string en el eje x e int en el eje y!

La combinación de tupla + switch es muy potente y concisa, y elimina toda una categoría de errores. Juntos, ellos limpiamente listan todas las posibles combinaciones de valores. Una tupla del tipo (bool, bool) efectivamente tiene 2 * 2 = 4 posibilidades, y el sistema de tipos te pide que cubras todos los 4. Esto se presta bien para refactorizar código; en lugar de agregar informalmente algunos if-else en valores arbitrarios aquí y allá, puedes señalar directamente la rama exacta del switch que necesitas alterar, nada más y nada menos.

Decisiones de Diseño

La existencia de tuplas puede parecer extraña para aquellos que vienen de lenguajes sin tipos. "¿Por qué no usar solo un array/lista?"

Un sistema de tipos no es todopoderoso, ni debería serlo; algunas concesiones deben ser aplicadas con el fin de mantener el lenguaje simple, eficaz (velocidad de compilación y en ejecución) y fácil de entender. Las listas en Reason, por ejemplo, son más flexibles en tamaño; pueden ser encadenadas, añadidas, divididas, etc. En cambio, tienen que ser homogéneas (únicamente pueden contener un solo tipo de valor por lista), y un acceso aleatorio indexado sobre ellos podría no ser válido *. Las tuplas, por otra parte, a través de su tamaño restringido, es más rápido, le da al sistema de tipos el margen de maniobra para rastrear exhaustivamente todos sus tipos de elementos y garantiza un acceso seguro. En general, notaras algunas prominentes, concesiones elegantes en un sistema de tipos: los campos de registros son fijos pero puede ser heterogéneos, mientras que los campos de un mapa son flexibles pero homogéneos, etc.

Una tupla en Reason es tipada "estructuralmente". Esto significa que incluso si no anotas tus datos con un tipo explícito, el compilador puede todavía deducirlo mirando su contenido, su uso, etc. Mientras las declaraciones y las formas de uso inferidas coincidan, ¡todo irá bien!

\ * No es que el sistema de tipos de Reason no puede aceptar listas heterogéneas, dimensionadas dinámicamente; ¡realmente puede (sugerencia: GADT)! Pero hacer tal característica por defecto incrementa el aprendizaje inicial y la comprensión del código. ¡Solo por que los tipos pueden lograrlo no significa que siempre es una buena idea dejar algunos trozos de código crecer ilimitadamente en complejidad!
