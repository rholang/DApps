Tuple

Tuplas são

  • imutáveis
  • ordenadas
  • possuem tamanho fixo definido em tempo de criação
  • heterogêneas (podem conter valores de tipos diferentes)
let ageAndName = (24, "Lil' Reason");
let my3dCoordinates = (20.0, 30.5, 100.0);

Tipos de tuplas podem ser usados como anotação de tipos também. Tipos de tuplas se parecem visualmente com tuplas de valores.

let ageAndName: (int, string) = (24, "Lil' Reason");
/* a tuple type alias */
type coord3d = (float, float, float);
let my3dCoordinates: coord3d = (20.0, 30.5, 100.0);

Nota: não existe tupla de tamanho 1. Você apenas deve utilizar o próprio valor, quando esse for o caso.

Uso

A biblioteca padrão oferece fst e snd (aqui, sob "Operações de par"), essas funções de conveniência permitem que você acesse o primeiro e o segundo elemento de uma tupla de 2 elementos. Geralmente, você acessa uma tupla de n elementos utilizando "destructuring" (descrita mais a frente na barra lateral):

let (_, y, _) = my3dCoordinates; /* now you've retrieved y */

O _ indica que você está ignorando o membro indicado da tupla.

Tuplas não são utilizadas para atualização mutáveis; você deve criar uma nova tupla desestruturando a antiga.

Dicas & Truques

Você usa tuples em situações onde é útil passar múltiplos valores sem muita cerimônia. Por exemplo, para retornar muitos valores:

let getCenterCoordinates = () => {
  let x = doSomeOperationsHere();
  let y = doSomeMoreOperationsHere();
  (x, y)
};

Ou para verificar a ocorrência de padrões (descrito mais tarde) na conjunção de possibilidades:

switch (isWindowOpen, isDoorOpen) { /* this is a 2-tuple */
| (true, true) => ...
| (true, false) => ...
| (false, true) => ...
| (false, false) => ...
}

Tente manter o uso de tuplas local. Para estruturas de dados que tem um tempo de vida maior e são passadas frequentementes entre funções, prefira um record, que possue campos com nomes.

Um tipo de tupla pode ser chamado também de "produto de tipo", e (string, int) é escrito como string * int em alguns lugares. A ideia é que uma tupla seja realmente um "produto cartesiano"; Imagine uma grade 2D, com string no eixo x e int no eixo y!

A combinação de tuplas + switch é bastante poderosa e concisa, e erradica toda uma categoria de bugs. Juntos, eles listam de maneira clara todas as combinações de valores possíveis. Uma tupla do tipo (bool, bool) de fato possuí 2 * 2 = 4 possibilidades, e o sistema de tipos pede para você cobrir as 4. Isso se mostra ótimo para refatorações de código; ao invés de adicionar informalmente alguns if-else com valores arbitrários aqui e ali, você pode localizar de maneira direta e exata a combinação do switch que você precisa alterar, nem mais, nem menos.

Decisões de Design

A existência de tuplas pode parecer estranha para aqueles que vem de linguagens sem tipos. "Por que não usar uma lista ou array apenas?"

Um sistema de tipos não é todo-poderoso, nem deveria ser; alguns tipos de trade-offs precisam ser aplicados de maneira a manter a linguagem simples e performática (tanto para compilação quanto para velocidade de execução) e simples de entender. As listas de Reason, por exemplo, são mais flexíveis no tamanho; elas podem ser concatenadas, anexadas, repartidas, etc. Em troca, elas precisam ser homogêneas (podem conter apenas um tipo de valor por lista), e acessos por indíce aleatório nelas podem nem sempre ser válidos*. Tuplas, por outro lado, através da sua restrição no tamanho, é mais rápida, e dá ao sistema de tipos margem para exaustivamente verificar todos os tipos dos items que ela armazena, e garante acesso seguro. De maneira geral, você provavelmente perceberá poucos sabores de trade-offs proeminentes em um sistema de tipos: campos de records são fixos porém podem ser heterogêneos, enquanto campos de um map são flexiveís mas homogêneos, etc.

Uma tupla em Reason é tipada "estruturalmente". Isso quer dizer que mesmo se você não anotar seu dado com um tipo explícito, o compilador ainda pode deduzir qual tipo é olhando no seu conteúdo, seu uso, etc. Enquanto as declarações e as formas inferidas dos usos combinarem, está tudo bem!

* Não é que o sistema de tipo Reason não pode aceitar listas heterogêneas e de tamanho dinâmico; na verdade pode (dica: GADT)! Mas fazer esta característica o padrão aumenta tanto a sobrecarga na curva de aprendizado da linguagem quanto na compreensividade do código. Não é apenas porque os tipos podem fazer isso não quer dizer que é sempre uma boa ideia deixar pedaços do código crescerem com complexidade sem limites!
