Jogando Javascript no meio do seu código Reason

If you're just hacking things together, this can be very nice, but you also have all of the unsafety of JavaScript code

Js.log("this is reason");

[%bs.raw {| console.log('here is some javascript for you') |}];

{| e |} são os delimitadores de strings multi-linha em OCaml. Você também pode colocar uma tag ali, ex. {algo| e então ela irá procurar por um |algo} correspondente para fechamento.

E segue aqui o JavaScript resultante:

// Generated by BUCKLESCRIPT VERSION 1.7.4, PLEASE EDIT WITH CARE
'use strict';
console.log("este é o reason");
console.log('aqui está um pouco de javascript para você');

Inserindo Javascript no seu código e tornando-o acessível pelo Reason

E se você quiser um valor que possa ser usado por seu código Reason?

Js.log("this is reason");
let x = [%bs.raw {| 'here is a string from javascript' |}];
Js.log(x ++ " back in reason land"); /* ++ is the operator for string concat */

Agora você deve estar pensando "que mágica é essa?? How did ocaml know that x was a string?" It doesn't. The type of x in this code is a magic type that will unify with anything! This is quite dangerous and can have cascading effects in OCaml's type inference algorithm.

let y = [%bs.raw {| 'something' |}];
Js.log(("a string" ++ y, 10 + y));
/* perigo!! OCaml não irá impedi-lo de usar y como tendo 2 tipos totalmente distintos */

Para consertar isso, você deve sempre prover um tipo concreto para o resultado de um bs.raw.

let x: string = [%bs.raw {| 'bem-tipada' |}];
Js.log(x ++ "de volta na terra do Reason");
/* OCaml acusará um erro caso você tente usar x como qualquer outra coisa além de string */

E aqui está a saída!

// Generated by BUCKLESCRIPT VERSION 1.7.4, PLEASE EDIT WITH CARE
'use strict';
console.log("este é o reason");
var x = ( 'aqui está uma string do JavaScript' );
console.log(x + " de volta na terra do Reason");
var y = ( 'algo' );
console.log(/* tupla */[
      "uma string" + y,
      10 + y | 0
    ]);
var x$1 = ( 'bem-tipada' );
console.log(x$1 + "de volta na terra do Reason");

A diferença entre os 2 %% da seção anterior e este 1 % aqui é importante! [%%something ...] é um "ponto de extensão" de OCaml que representa uma declaração no escopo superior (ele não pode aparecer dentro de uma função ou valor, por exemplo). [%something ...]é um "ponto de extensão" que representa uma expressão, e pode ser colocado em praticamente qualquer lugar -- mas garanta que o JavaScript que você coloca dentro é realmente uma expressão! Ex. não coloque ponto e vírgula após ela, ou você obterá um erro de sintaxe quando tentar rodar o JavaScript resultante.

Inserindo em uma função & passando valores

Nós precisaremos de um pouco de conhecimento sobre a representação de BuckleScript dos vários valores em tempo de execução para fazer isso funcionar.

  • strings são strings, ints e floats são somente números
  • um Array é uma lista mutável de tamanho fixo em OCaml, e é representada como um simples array JavaScript.
  • um List é uma lista encadeada imutável em estilo funcional, e é definitivamente mais idiomática para a maioria dos casos de uso. However, its representation is more complicated (try Js.log([1,2,3,4]) to check it out). Por causa disso, eu geralmente converto de & para Arrays quando eu estou falando com JavaScript, via Array.of_list e Array.to_list.
  • Se você quiser se aprofundar, existe uma lista na documentação de BuckleScript

Sabendo disto, nós podemos escrever uma função em JavaScript que aceita um array e retorna um número, sem grandes problemas.

let calcularComJs: (array(int), int) => int = [%bs.raw
  {|
 function (numeros, fatorDeEscala) {
   var resultado = 0;
   numeros.forEach(numero => {
     resultado += numero;
   });
   return resultado * fatorDeEscala;
 }
|}
];

let calcular = (numeros, fatorDeEscala) => calcularComJs(Array.of_list(numeros), fatorDeEscala);

Js.log(calcular([1, 2, 3], 10)); /* -> 60 */

Obviamente, essa função que escrevi em JavaScript poderia ser portada para Reason sem muita inconveniência.

Lembre que isso é uma "saída de emergência" que tem utilidade considerável no aprendizado já que permite você comece rapidamente a fazer alguma coisa, mas é um bom exercício retornar e converter estas coisas para código Reason seguro.

I've run into more than a few bugs because of raw JavaScript that I added to save time

Estabelecendo-se e disciplinando-se

Até agora nós temos usado bs.raw, que é uma maneira rápida e vaga de fazer as coisas, e não é recomendada para produção.

Mas e se realmente precisarmos chamar uma função que está em código JavaScript? Isso é necessário para interagir com DOM ou usar módulos do Node. Em BuckleScript, você usa uma declaração external (documentação).

Obter um valor e obter uma função são ambos bem simples:

[@bs.val] external pi : float = "Math.PI";
let tau = pi *. 2.0;
[@bs.val] external alert : string => unit = "alert";
alert("olar");

Mas e no caso de precisarmos de algo mais complexo? Aqui está como poderíamos chamar getContext em um nó Canvas do DOM:

type canvas;

type context;

/* we're leaving these types abstract, because we won't
 * be using them directly anywhere */
[@bs.send] external getContext : (canvas, string) => context = "getContext";

let myCanvas: canvas = [%bs.raw {| document.getElementById("mycanvas") |}];

let ctx = getContext(myCanvas, "2d");

Vamos desembaraçar o que está ocorrendo aqui. Nós criamos alguns tipos abstratos para o nó Canvas do DOM e o objeto RenderingContext associado.

Então, nós criamos uma função getContext, mas ao invés de @bs.val nós usamos @bs.send, e usamos uma string vazia para o texto do external. @bs.send significa "nós estaremos chamando um método do primeiro argumento", que neste caso é o canvas. Dado o trecho acima, BuckleScript vai traduzir getContext(theFirstArgument, theSecondArgument) para theFirstArgument.getContext(theSecondArgument, ...).

A string vazia significa "o nome em JS é o mesmo que o nome que estamos dando ao external no domínio do BuckleScript" – neste caso getContext. Se nós desejássemos dar um outro nome (como getRenderingContext), aí necessitaríamos prover a string "getContext" para que o BuckleScript chame a função correta.

Vamos só adicionar mais uma função para tornar as coisas mais interessantes.

[@bs.send] external fillRect : (context, float, float, float, float) => unit = "fillRect";

Agora podemos desenhar!

fillRect(ctx, 0.0, 0.0, 100.0, 100.0);

Não é muito, mas adicionar outros métodos do canvas é similar, e aí você pode começar a fazer algumas coisas muito legais.

E como fica a aparência do JavaScript compilado?

'use strict';
var tau = Math.PI * 2.0;
alert("hello");
var myCanvas = ( document.getElementById("mycanvas") );
var ctx = myCanvas.getContext("2d");
ctx.fillRect(0.0, 0.0, 100.0, 100.0);

Eita! Viu como BuckleScript simplesmente inseriu nossa variável pi para nós? E a saída parece quase como escrita à mão.

Using bibliotecas JavaScript existentes

When folks write wrappers for a particular JavaScript library, they'd usually publish it to npm. Head over to the Libraries to find out how to find these.

To use a library that does not have existing wrappers, however, you'll want to first install the npm package as usual, e.g. using npm install --save <package-name>, then just go ahead and write your wrapper. Você provavelmente vai achar o recurso de FFI do bs.module particularmente útil; ele emite os imports ou requires adequados ao formato de módulo do alvo da compilação do JS.

As an example, here's the entire source code of the bs.glob wrapper (converted to Reason, the original is OCaml):

type error;

[@bs.module] external glob : (string, (Js.nullable(error), array(string)) => unit) => unit = "glob";

[@bs.val] [@bs.module "glob"] external sync : string => array(string) = "glob";

E as partes relevantes do package.json:

{
  "name": "bs-glob",
  "version": "0.1.0",
  ...
  "devDependencies": {
    "bs-platform": "^1.9.1"
  },
  "dependencies": {
    "glob": "^7.1.2"
  }
}
