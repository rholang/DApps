Project Structure

Essas são as práticas comunitárias existentes, não codificadas, que atualmente são propagadas por um acordo informal. Nós podemos remover algumas delas e reforçar outras em algum ponto. Neste momento, elas são apenas recomendações para a facilidade dos recém-chegados.

Nomenclatura de Arquivos

Nomes de arquivos capitalizados (primeira letra em maiúsculo).

Justificativa: Os nomes de módulos só podem ser capitalizados. Recém-chegados perguntam com frequência como um arquivo é mapeado para um módulo, e por que draw.re mapeia para o módulo Draw, e algumas vezes tentam referenciar um módulo através de identificadores não capitalizados. Usando Draw.re torna esse mapeamento mais direto. Isso ajuda também alguns nomes de arquivos que ficariam estranhos na forma não capitalizada: uRI.re.

Ignore o arquivo .merlin

Ele é gerado pelo build system(pelo menos para o workflow JS) e você não precisa edita-lo manualmente. Também tente não enviar para o repositório.

Justificativa: .merlin é para que o Merlin entenda o layout do projeto e possar prover o ferramental do editor de texto. O arquivo contém caminhos absolutos, que inclusive não são multiplataforma (por exemplo: os caminhos do Windows são diferentes).

Diretórios

Try not to have too many nested folders. Keep your project flat, and have fewer files (reminder: you can use nested modules).

Justification: The file system is a tree, but your code's dependencies are a graph. Because of that, any file & folder organization is usually imperfect. While it's still valuable to group related files together in a folder, the time wasted debating & getting decision paralysis over these far outweight their benefits. We'll always recommend you to Get Work Done instead of debating about these issues.

Dependências de Terceiros

Mantenha elas ao mínimo.

Justificativa: Uma linguagem compilada e tipada estaticamente não pode modelar suas dependências facilmente como pode uma linguagem dinâmica, especialmente quando estamos utilizando NPM/Yarn (para reduzir a curva de aprendizagem a médio prazo), ambos não foram feitos levando em conta sua utilização com o Reason/BuckleScript. Manter suas dependências simples e enxutas ajuda a reduzir a possibilidade de conflitos (por exemplo: dependências em "diamente" ou interfaces conflitantes).

Modele dependências de terceiros como dependencies, não peerDependencies.

Justificativa: você pode tratá-las contextualmente como detalhes de implementação que podem ser alterdas um dia.

Documentação

Os tenha. Gasta-se mais esforço tornando-os bons (exemplos, armadilhas) e profissionais que tornando-os apenas bonitinhos. Use exemplos e evite usar nomes como foo e bar. Sempre há nomes mais concretos (este é um exemplo, não precisa ser abstrato/generalizado. A documentação da API o fará plenamente). Para postagens em blogs, não repita a documentação exatamente, descreva as transições do velho para o novo e o motivo (por exemplo: "era um componente, agora é uma função, pois ...").

Justificativa: é difícil para iniciantes distinguir entre uma biblioteca simples/decente e uma que é extravagante. Pelo bem da comunidade, não tente demasiadamente concorrer com o código alheio. Espalhe a palavra, mas também use o seu julgamento.

PPX & Outras Meta Ferramentas

Mantenha elas ao mínimo. PPX, desde que usado em casos decentes (geração de impressões, acessores e serializadores/desserializadores), pode ajudar acausar uma grande queda na curva de aprendizado dos novatos; no pico da sintaxe, semântica, tipos, ferramentas de build e FFI que eles têm que aprender, aprender transformações específicas do código de uma biblioteca é um passo extra. Macros mais invasivos fazem com que o próprio código seja menos semanticamente significante, uma vez que a essência está escondida em outro lugar.

Paradigmas

Não abuse das características chiquérrimas da linguagem. Deixe espaço para futuras APIs, mas não sobrecarregue a arquitetura das coisas.

Justificativa: código simples ajuda novatos a entenderem e potencialmente contribuírem para o seu código. Contribuir é a melhor maneira para eles aprenderem. A ajuda extra que você pode receber pode também se sobrepor ao ganho utilizando uma característica mais inteligente da linguagem. Tente novas formas de resolver os problemas em projetos mais casuais! Você pode descobrir novas maneiras de arquitetar o código.

Publicação

If it's a wrapper for a JS library, don't publish the JS artifacts. Se é uma bilioteca própria, publique os artefatos em lib/js se você crê que consumidores de javascript podem utilizá-la. Este é especialmente o caso em que gradualmente converte-se uma biblioteca em javascript para Reason + BuclkeScript enquanto não quebra consumidores de JS existentes.

Do put the keywords "reason" and "bucklescript" in your package.json keywords field. This allows us to find the library much more easily for future purposes.

Justificativa: Seja legal com os usuários da sua biblioteca JS. Eles serão futuros Reasoners.
