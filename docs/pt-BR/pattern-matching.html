Pattern Matching!

Make sure you've read up on Variant first.

Finalmente chegamos aqui! Pattern matching é um dos melhores recursos da linguagem. É como desestruturação, mas obtendo ainda mais auxílio do sistema de tipos.

Uso

Considere um variante:

type resultado =
  | ResultadoRuim(int)
  | ResultadoBom(string)
  | SemResultado;

Ao usar a expressão switch nele, você pode "desestruturá-lo":

let dados = ResultadoBom("Produto enviado!");

let mensagem =
  switch (dados) {
  | ResultadoBom(aMensagem) => "Sucesso! " ++ aMensagem
  | ResultadoRuim(codigoDeErro) => "Algo está errado. O código de erro é: "++ string_of_int (codigoDeErro)
   };

Note como desestruturamos data ao mesmo tempo que tratamos cada uma das variações. O switch no trecho acima vai gerar um alerta do compilador:

Warning 8: this pattern-matching is not exhaustive.
Here is an example of a value that is not matched:
NoResult

Não é ótimo? Enquanto checava o formato do nosso dado, o sistema de tipos nos alertou de uma variação não tratada. Este aspecto condicional é o que diferencia pattern matching da simples desestruturação. A maioria das estruturas de dados com um aspecto "se isso então aquilo" tomam proveito funcionalidade:

switch (minhaLista) {
| [] => print_endline("Lista vazia")
| [a, ...oResto] => print_endline("lista com valor de cabeça " ++ a)
};

switch (meuArray) {
| [|1, 2|] => print_endline("Este é um array com os itens 1 e 2")
| [||] => print_endline("Este é um array sem elementos")
| _ => print_endline("Este é um array")
};

O caso _ é um caso especial "pega-tudo" que permite que casos não considerados nas alternativas anteriores sejam processados neste ramo.

Você pode inclusive usar switch com strings, inteiros e outros tipos. Você pode até mesmo ter várias formas de dado direcionadas a um mesmo resultado!

let resposta =
  switch (mensagem) {
  | "Reason é bem legal" => "Pois é"
  | "boa noite" => "Até mais!"
  | "olá" | "oi" | "eae" | "opa" => "oi pra você também!"
  | _ => "Bom te conhecer!"
  };

Combinado com outras estruturas de dados, pattern matching pode resultar em código extremamente conciso, verificado pelo compilador e com grande performance:

let mensagem = 
  switch (dados) {
  | ResultadoBom(aMensagem) => "Sucesso! " ++ aMensagem
  | ResultadoRuim(0 | 1 | 5) => "Algo está errado. É um problema com o servidor."
  | ResultadoRuim(codigoDeErro) => "Erro desconhecido ocorreu. Code: " ++ string_of_int(errorCode)
  | NoResult => "Things look fine"
  };

Observação: você pode passar somente literais (i.e. valores concretos) como um formato de dado, e não nomes definidos com let binding e outras coisas. O trecho a seguir não funciona como esperado:

let minhaMensagem = "Olá";
switch (saudacao) {
| minhaMensagem => print_endline("Olá para você")
};

Ao contrário, ele estará correspondendo a qualquer string, e atribuindo esta ao nome myMessage naquele ramo do switch, o que não é o resultado almejado.

Cláusulas When (Quando)

Quando você realmente precisar de alguma lógica arbitrária num pattern match simples, você inserir algumas cláusulas when, que são basicamente açúcar sintático para if:

let mensagem =
  switch (dados) {
  | ResultadoBom(aMensagem) => ...
  | ResultadoRuim(codigoDeErro) when erroDeServidor(codigoDeErro) => ...
  | ResultadoRuim(codigoDeErro) => ... /* se não for erro de servidor */
  | SemResultado => ...
  };

Match em Exceções

If the function throws an exception (covered later), you can also match on that, in addition to the function's normally returned values.

switch (List.find((i) => i === theItem, myItems)) {
| item => print_endline(item)
| exception Not_found => print_endline("No such item found!")
};

Padrões aninhados

Barras | aninhadas funcionam como esperado:

switch (student) {
| {name: "Jane" | "Joe"} => ...
| {name: "Bob", Job: Programmer({fullTime: Yes | Maybe})} => ...
};

Padrões em Todo Lugar

Você pode colocar um padrão em todo lugar onde colocaria uma "declaração de variável" normal:

type leftOrRight =
  | Left(int)
  | Right(int);

let i = Left(1);

/* magic! */
let Left(v) | Right(v) = i;

Dicas & Truques

Desaninhe seus casamentos de padrões sempre que possível. Esse é um jeito confiável de remover bugs. Exemplo abaixo.

Não use o caso "pega-tudo" _ demais. Ele impede que o compilador avise que você esqueceu de cobrir um caso (checagem de exaustividade), o que seria especialmente útil após uma refatoração onde um novo caso foi adicionado a uma variante. Tente usar _ apenas quando houver infinitas possibilidades. Ex: string, int, etc.

Segue uma série de exemplos, de pior para melhor:

let optionBoolToBool = opt =>
  if (opt == None) {
    false;
  } else if (opt == Some(true)) {
    true;
  } else {
    false;
  };

Isso é simplesmente bobo =). Vamos transformar em casamento de padrões:

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(a) => a ? true : false
  };

Levemente melhor, mas ainda aninhado. O casamento de padrões permite que você faça isto:

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(true) => true
  | Some(false) => false
  };

A aparência ficou muito mais linear! Agora, você pode ser tentado a fazer o seguinte:

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(true) => true
  | _ => false
  };

Which is much more concise, but kills the exhaustiveness check mentioned above; refrain from using that. This is the best:

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(trueOrFalse) => trueOrFalse
  | None => false
  };

Muito difícil fazer um erro no código nesse ponto! Sempre que você pensar em usar um if-else com vários, ramos, prefira casamento de padrões ao invés disso. É mais conciso, além de eficiente também.

Decisões de Design

O famoso problema FizzBuzz estranhamente faz algumas pessoas "empacarem", parcialmente devido à sua natureza de paralisar programadores que buscam simplificar ou unificar os ramos condicionais em busca de elegância onde não a há. Esperamos que seja possível ver que, em geral, a concisão visual do casamento de padrões nos permite evitar a paralisia de decisão, simultaneamente mantendo todas as vantagens (e mais, como você viu) de usar vários if-elses. Não há nada errado com explicitamente listar todas as possibilidades; Casamento de padrões corresponde, na matemática, à análise de casos, uma técnica válida de resolução de problemas que costuma ser extremamente conveniente.

Usar um switch do Reason pela primeira vez pode fazer você sentir que estava perdendo algo todos esses anos. Cuidado, ele pode estragar outras linguagens para você =).

Se você já tentou refatorar um if-else grande e aninhado, você pode ter percebido que é muito difícil acertar a lógica. Por outro lado, o uso casamento de padrões + tuplas mapeia conceitualmente para uma tabela 2D, em que cada célula pode ser independentemente preenchida. Isso garante que, sempre que você precisa adicionar um caso no switch, você pode atacar aquela e apenas aquela célula da tabela, sem quebrar outras células.

type animal = Cachorro | Gato | Passaro;
let resultado = switch(eGrande, meuAnimal) {
 | (true, Cachorro) => 1
 | (true, Cat) => 2
 | (true, Passaro) => 3
 | (false, Cachorro | Gato) => 4
 | (false, Passaro) => 5
};
