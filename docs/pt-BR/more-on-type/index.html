More on Type

Argumentos em Tipos!

Tipos podem aceitar parâmetros, semelhantes aos generics em outras linguagens. É como se um tipo fosse uma função que recebe argumentos e retorna um novo tipo! Os parâmetros precisam iniciar com '.

O caso de uso de um tipo parametrizado é acabar com duplicações. Antes:

/* essa é uma tupla de 3 items, explicações a seguir */
type intCoordinates = (int, int, int);
type floatCoordinates = (float, float, float);

let buddy: intCoordinates = (10, 20, 20);

Depois:

type coordinates('a) = ('a, 'a, 'a);

/* aplica a "função de tipo" coordinates e retorna o tipo (int, int, int) */
type intCoordinatesAlias = coordinates(int);

let buddy: intCoordinatesAlias = (10, 20, 20);

/* ou, mais comumente, escreva na mesma linha */
let buddy: coordinates(float) = (10.5, 20.5, 20.5);

Na prática, tipos são inferidos para você. Assim, a versão mais concisa do exemplo acima não poderia ser nada além de:

let buddy = (10, 20, 20);

O sistema de tipos infere que o tipo é (int, int, int). Nada mais precisa ser escrito.

Argumentos em tipos aparecem em todo lugar.

/* inferido como `list(string)` */
let greetings = ["hello", "world", "how are you"];

Se os tipos não aceitassem parâmetros (ou seja, se não tivéssemos "type functions"), a biblioteca padrão precisaria definir os tipos listOfString, listOfInt, listOfTuplesOfInt, etc.

Tipos podem receber mais de um argumento, e são agregáveis.

type result('a, 'b) =
  | Ok('a)
  | Error('b);

type myPayload = {data: string};

type myPayloadResults('errorType) = list(result(myPayload, 'errorType));

let payloadResults: myPayloadResults(string) = [
  Ok({data: "hi"}),
  Ok({data: "bye"}),
  Error("Something wrong happened!")
];

Tipos Mutualmente Recursivos

Da mesma forma que funções, tipos podem ser mutualmente recursivos através de and:

type student = {taughtBy: teacher}
and teacher = {students: list(student)};

Note que não há ponto e vírgula terminando a primeira linha e nenhum type na segunda linha.

Decisões de Design

Um sistema de tipos permitindo argumentos em tipos está basicamente permitindo funções no nível dos tipos. list(int) é na verdade a função de tipo list recebendo o tipo int, e retornando o tipo final, concreto, que você usaria em vários lugares. Você deve ter notado que em outras linguagens, isso é basicamente chamado de "generics". Por exemplo, ArrayList<Integer> em Java.

The principle of least power (princípio do menor potencial) se aplica quando você quer simplesmente quer "Fazer as Coisas na Prática". Se o domínio do problema permite, definitivamente escolha a solução menos abstrata (aka, a mais concreta) disponível, com isso a solução é alcançada mais rápido e possui menos indireções instáveis para você acompanhar. Por exemplo, prefira tipos a dados sem forma, prefira configuração orientada a dados a chamadas a funções que são turing-completas, prefira chamadas de funções a macros, prefira macros a bifurcações no projeto, etc. Quando você restringe seu domínio e poder, as coisas se tornam mais fáceis de analisar. Isto é, se o domínio é restrito o suficiente para permitir isso.

When a type system is an all-encompassing aspect of your program, we need to make sure we leave enough power in order not to overly constrain your expressiveness; without "type functions", you'd end up with quite a bit of boilerplate, e.g. hard-coded listOfInt, listOfString, listOfArrayOfFloat, their respective helper functions, etc. Porém, tome cuidado para não abusar do poder concedido a você através de um sistema de tipos consideravelmente poderoso. Em algumas ocasiões, está bem ter um pouco de verborragia para reduzir a necessidade por tipos que em contrário seriam "superpoderosos". Em último caso, balanceamento e trocas feitas com bom senso mostram seu pragmatismo e capacidade de julgamento mais do que tipos chiques!
