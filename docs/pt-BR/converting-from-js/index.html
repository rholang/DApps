Converting from JS

Preparação

Antes de prosseguir, tenha certeza que Reason é o que seu time necessita! Enquanto nós atestamos a popularidade de Reason e BuckleScript, não crie discussões desnecessárias com seus colegas, deixando uma má primeira impressão. Isto é difícil de corrigir posteriormente.

Este guia cobre um fluxo de trabalho que tem nos ajudado a converter os programas rápida e eficientemente. Não foi feito para abordar características de linguagem e FFI (apesar disso, coloca-as no contexto). É assumido um conhecimento básico de Reason e BuckleScript.

Sintaxe

Objetivo: antes de mais nada, torne a sintaxe do arquivo válida. Ignore tipos errados, módulos ausentes, má organização de arquivos, dependências externas em excesso, etc. Nós iremos retornar para corrigir estes problemas após estabelecer o teste de regressão "sem erros de sintaxe".

Dado que a sintaxe de Reason se assemelha o suficiente a de JavaScript, ao invés de iniciar um arquivo Reason em branco, simplesmente copie o arquivo js existente e trabalhe a partir dele.

Dica: não esqueça que você pode usar refmt no seu editor/terminal! Se você não souber, por exemplo, a precedência de algumas operações, coloque em volta tantos parênteses quantos quiser, e aí aplique refmt ao seu código e veja quais são mantidos. Da mesma forma, não há necessidade de perder tempo em indentação e espaçamentos; refmt cuida disso.

/* arquivo JS original que você copiou */
const school = require('school');

const defaultId = 10;

function queryResult(usePayload, payload) {
  if (usePayload) {
    return payload.student
  }
  return school.getStudentById(defaultId);
}

Segue algumas coisas que você faria nessa etapa:

  • Converta a sintaxe para chamada de função.
  • Converta var e const para let.
  • Apague os requires.
  • Faça outras modificações semelhantes. Para trechos que não tem um equivalente em BuckleScript, use bs.raw (documentação).

Novamente, se preocupe somente em tornar o arquivo sintaticamente válido. Tentar aprender ao mesmo tempo 1) sintaxe, 2) tipos e 3) outras semânticas ao mesmo tempo que se faz a conversão de um arquivo reduzirá sua velocidade de iteração a menos de um terço.

/* conversão sintaticamente válida, semanticamente incorreta */
/* const school = require('school'); */
let defaultId = 10;

let queryResult = (usePayload, payload) =>
  if (usePayload) {
    payload.student
  } else {
    /* não há necessidade de um return prematuro em Reason, if-else é uma expressão */
    school.getStudentById(
      defaultId
    )
  };

Tipos, passo 1

Objetivo: corrigir os tipos, mas só o suficiente para passar à próxima etapa.

Você irá ocasionalmente esbarrar em erros de sintaxe, porém não tão drásticos quanto os dos passos anteriores.

  • Mude foo.bar para foo##bar. Esse recurso "saída de emergência" do BuckleScript será seu amigo a médio prazo.
  • Converta {foo: bar} para [%bs.obj {foo: bar}] (documentação). Após usar o refmt, isso será reescrito como {"foo": bar}.
  • Para se comunicar com arquivos JS externos, use external. No BuckleScript, isso é conhecido como foreign function interface.
    • Traga externals para o seu arquivo. Não há necessidade de criar arquivos limpos e bem separados para dependências externas agora. Iremos voltar a isso.
    • Se é trabalhoso demais tipar corretamente a entrada e saída de um external, use alguns tipos polimórficos como substitutos, ex: external estudantePorId: 'tantoFaz => 'tambemTantoFaz = ....
    • Para tipos de dados & padrões que são difíceis de converter corretamente, vocêo pode ocasionalmente criar conversores como external transformacaoInsegura : meuTipoDeDados => outroTipoDeDados = "%identity";.

Este é a primeira passagem; ao final os tipos provavelmente serão diferentes. Por enquanto, colha os frutos! Uma vez que você termine de consertar todos os erros de tipagem, seu arquivo JS deverá ser gerado. Mantenha-o aberto lado a lado. É chegado o momento de retornar e arrumar as gambiarras!

/* sintaticamente válido, ainda incorreto semanticamente, porém melhor */
[@bs.module "school"] external getStudentById : 'whatever => 'whateverElse = "getStudentById";

let defaultId = 10;

let queryResult = (usePayload, payload) =>
  if (usePayload) {
    payload##student /* isto será inferido como `Js.t 'a` */
  } else {
    getStudentById(defaultId)
  };

Semântica de Execução

Objetivo: consertar os erros no JS gerado na saída.

Compare-o com seu arquivo JS inicial. A saída provavelmente está incorreta; você provavelmente cometeu erros ao traduzir alguns idiomas e alguns externals.

  • Dê tipo ao formato dos objetos JS (os trechos que exigiram ##).
  • Converta qualquer uma dessas ocorrências para records/variants/outros tipos idiomáticos em OCaml.

Durante essa parte, confira a saída para cada alteração.

type student; /* tipo abstrato, descrito posteriormente */

[@bs.module "school"] external getStudentById : 'whatever => student = "getStudentById";

type payloadType = {. "student": student};

let defaultId = 10;

let queryResult = (usePayload, payload: payloadType) =>
  if (usePayload) {
    payload##student
  } else {
    getStudentById(defaultId)
  };

Limpeza (Tipos, Passagem 2)

Objetivo: tornar seus tipos corretos (aka 'sound').

Retorne para consertar tudo que você ignorou durante a primeira passagem.

  • Tenha certeza que você não tem quaisquer tipos 'coisa deixados nos externals.
  • Você pode manter seus externals no seu arquivo ou movê-los para um arquivo a parte.
/* no arquivo atual */
type payloadType = {. "student": School.student}; /* TODO: mover isto para outro local! */

let defaultId = 10;

let queryResult = (usePayload, payload: payloadType) =>
  if (usePayload) {
    payload##student
  } else {
    School.getStudentById(defaultId)
  };

/* em um arquivo School.re dedicado */
type student;

[@bs.module "School"] external getStudentById : int => student = "getStudentById";
[@bs.module "School"] external getAllStudents : unit => array(student) = "getAllStudents";

O tipo student não tem conteúdo; isto é chamado de tipo abstrato (abstract type). Esta é uma maneira conveniente de descrever um relacionamento entre chamadas a código externo sem saber o real formato dos dados sendo passado.

Com isso, tá tudo certinho!

Dicas

Não tenta converter um arquivo JS em um arquivo Reason irretocável de uma vez só. Tal método provavelmente irá ser mais lento! Está ok ter externals e bs.obj remanescentes, e temporariamente não aproveitar recursos agradáveis de OCaml (variants, argumentos nomeados, etc.). Uma vez que você tenha convertido outros arquivos relacionados, você pode retornar e refatorar de maneira mais rápida tomando proveito do sistema de tipos.

Whatever nice utilities you find, put them in a tempUtils.re file or something. They're easy examples for your colleagues and removes some conversion churns.

Nós recomendamos fortemente que você realize o controle de versão da saída JS. Isto torna a integração com seu sistema de build quase transparente, e garante que quando você não está presente seus colegas ainda conseguem fazer pequenas mudanças, conferir as diferenças na saída e corrigir pequenos erros. Também é um ótimo argumento para o uso da tecnologia (principalmente para gerentes!) que a saída JS seja versionada e portanto diretamente modificável em caso de correções de emergência. Até mesmo em caso de atualização da versão do BuckleScript você será alertado de qualquer diferença na saída. É como o Jest snapshots, embutido!
